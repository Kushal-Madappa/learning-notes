## Today's work

- **LeetCode (2):**
  - **Diameter of Binary Tree (#543, Easy)** — find the longest path
    (in edges) between any two nodes. The **hybrid template** lives
    here: a single post-order DFS that *returns* the height of the
    current subtree while *mutating* an outer `self.best` whenever
    `left_height + right_height` exceeds it.
    - **Why O(n) and not O(n²):** The naïve "for each node, compute
      left height + right height, take the max" recomputes heights
      O(n) times. The hybrid template piggybacks the diameter update
      onto the same post-order traversal that's already computing
      heights, so each node does O(1) work. One pass.
    - **The "edges vs. nodes" gotcha:** Diameter is measured in edges,
      not nodes. If you define `height(None) = 0` and
      `height(leaf) = 0`, then `left + right` at any node gives the
      path length *through* that node in edges. If you accidentally
      use `height(None) = -1` to "model the missing edge," you're off
      by 2 at every leaf.
    - **Why the longest path may not pass through the root:** Imagine
      a tree where the left subtree is a long chain and the right
      subtree is just one node. The longest path lives entirely inside
      the left subtree. The hybrid template catches this because the
      update happens at *every* node, not just the root.
    - **Complexity:** O(n) time, O(h) recursion stack.
    - Verified: 2 LC samples + 7 hand-built edges (empty, single,
      V-shape, left chain, right chain, balanced, asymmetric depths,
      diameter-not-through-root). All 9 pass.

  - **Balanced Binary Tree (#110, Easy)** — check that every node's
    left/right subtree heights differ by at most 1. Introduces the
    **-1 sentinel trick**: a value-returning DFS where the return
    value is "height if balanced, -1 if unbalanced." The moment any
    subtree reports -1, every ancestor short-circuits and also
    returns -1.
    - **Why -1 is a safe sentinel:** Subtree heights are non-negative
      integers (`height(None)=0`, `height(leaf)=0`, etc.), so -1 can
      never collide with a real value. It unambiguously means "we
      already proved unbalance somewhere below." If your problem's
      value domain *could* include -1, you'd need a separate flag.
    - **Why this beats the naïve O(n²):** The textbook approach is
      "at each node, compute left height, right height, check `|L-R|
      <= 1`, recurse." Each `height()` call walks its whole subtree —
      O(n²) overall. The sentinel version does a single post-order
      pass: O(n).
    - **Why the sentinel doesn't slow the worst case:** Short-circuit
      returns only ever skip work. They never add cost. Worst case
      is still 1 × n = O(n).
    - **Complexity:** O(n) time, O(h) recursion stack.
    - Verified: 2 LC samples + 8 hand-built edges (empty, single,
      2-node, chains both directions, complete trees of depths 3 and
      4, deep-imbalance-under-balanced-root). All 10 pass.

## Key concepts

### 1. The hybrid template — "return one thing, mutate another"
The recursion toolkit for trees is now four-layered:

| Template | Helper returns | Helper mutates | Example |
|----------|---------------|----------------|---------|
| Pure-value | something (depth, count) | nothing | #104 Max Depth |
| Pure-mutation | nothing (in-place edit) | the tree (or shared state) | #226 Invert |
| Two-tree | bool / count | nothing | #100 Same, #101 Symmetric |
| **Hybrid** | something (height) | outer state (max diameter) | **#543 Diameter** |

The hybrid template is the one that unlocks "longest / largest /
deepest / widest path in a tree" problems. Once you see it once, you
see it everywhere: #124 Maximum Path Sum (return max-down-path,
update max-through-node), #687 Longest Univalue Path (return
matching-length-down, update longest-through-node), #1245 Tree
Diameter (general graph version of #543). The shape is always:

```python
def helper(node):
    if node is None: return BASE
    left  = helper(node.left)
    right = helper(node.right)
    self.best = max(self.best, COMBINE(left, right))   # mutation
    return EXTEND(left, right)                          # return
```

`COMBINE` is "what we measure *through* this node." `EXTEND` is "what
we hand up to the parent so it can keep computing." The two are
different — that's the whole point. The diameter through a node
(`left + right`) is not the same as the height handed up
(`1 + max(left, right)`).

### 2. The -1 sentinel trick — overload the return value
#110 introduces a sibling pattern to the hybrid template. Instead of
*mutating* outer state to signal "we found something bad," you
*overload the return value* with an impossible value (here, -1, since
real heights are non-negative). Every ancestor checks for the
sentinel and propagates it up.

```python
def helper(node):
    if node is None: return GOOD_BASE
    L = helper(node.left)
    if L == SENTINEL: return SENTINEL
    R = helper(node.right)
    if R == SENTINEL: return SENTINEL
    if not LOCAL_OK(L, R): return SENTINEL
    return COMBINE(L, R)
```

This is cheaper to set up than the hybrid template (no outer state
to manage) but only works when you can find an "impossible value" in
the return type's domain. For booleans you can't (`True` and `False`
are both valid), so you'd fall back to the hybrid template. For
counts, `-1` is almost always safe. For pointers, `None` works if
real nodes are non-None.

### 3. The "case-sensitive Yes/No" trap (CodeChef cakewalk reflex)
LECANDY and a whole family of CodeChef cakewalk problems demand
*exact case-sensitive* answers — "Yes" not "YES," "yes," or "YeS".
The problem statement makes it explicit, but it's the single most
common source of WA on cakewalk problems. The reflex from now on:

> **Copy the expected output string from the problem statement
> verbatim. Never retype it.**

If the problem says the answer is `Yes` or `No`, write those exact
strings as constants. If it says `INDIAN` / `NOT INDIAN` / `NOT
SURE`, all-caps. Don't rely on memory.

### 4. The two-pointer "shrink-from-both-ends" palindrome pattern
RAINBOWA looks complex but collapses to the same two-pointer pattern
as palindrome-check, with one twist: we shrink in *blocks of one
value at a time* rather than one element. For each v=1..6:
- Sweep `i` forward while `a[i] == v`, count.
- Sweep `j` backward while `a[j] == v`, count.
- Demand the two counts match.

After v=6, the only thing left between `i` and `j` is the 7-block at
the peak — verify it's all 7s and non-empty.

This generalizes nicely:
- **#125 Valid Palindrome** — shrink one char at a time, both ends
  must match.
- **#680 Valid Palindrome II** — same, with one "skip" allowed.
- **#1216 Valid Palindrome III** — same, with `k` skips allowed (DP
  or two-pointer with memoization).

The shape is always "two pointers, shrink from both ends, match a
local condition at each step, terminate when they meet (or cross)."
