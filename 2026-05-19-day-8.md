## Key concepts

### 1. Count-map vs first-index-map — the two prefix-sum dialects
Yesterday's note ended on the right intuition; today it's concrete.
Same skeleton, different bookkeeping, different question shapes:

| Dialect | What the map stores | Question it answers |
|---------|---------------------|---------------------|
| **Count map**       | `key -> # of prior prefixes with this key` | *How many* subarrays satisfy ... |
| **First-index map** | `key -> earliest index where this key appeared` | *Does there exist* ... and *what is the longest* ... |

LC 560 / 974 are count-map problems ("how many subarrays sum to k /
are divisible by k"). LC 523 / 525 are first-index problems ("is
there a length-≥-2 subarray divisible by k?", "longest balanced
subarray?"). The map type follows directly from whether you're
**counting** or **bounding length**.

### 2. The `{0: -1}` sentinel — why both dialects need a seed, but for different reasons
- In the count map, `counts[0] = 1` so that a prefix which is itself
  the target gets counted (the empty left-prefix `P[0] = 0`).
- In the first-index map, `first_idx[0] = -1` so that a prefix which
  is itself the target yields `length = i - (-1) = i + 1` — the
  whole-prefix length, not `i` (off-by-one trap).

Same value, two different mental models. Skipping the seed is the
single most common bug in this family.

### 3. Why we **don't** overwrite in the first-index map
When we see the same key again, we *consult* the map but we **leave
it alone**. The whole point of "first index" is the *earliest* index;
overwriting would shrink the candidate window every time we revisit
a key, making longest-subarray queries silently wrong. This is a
deliberate asymmetry vs. the count map (which is updated every step).

### 4. Transform-then-prefix (LC 525)
`0 → -1` is the standard trick that re-expresses a *balance*
question ("equal 0s and 1s") as a *sum* question. Once it's a sum
question, prefix-sum machinery applies verbatim. The general move:

    "balance / parity / equality" question
        --(suitable ±1 / mod-2 / XOR transform)-->
    "sum == 0" or "sum % m == 0" question
        --(prefix-sum + map)-->
    O(n) answer.

This is the *single most reusable* idea in the whole prefix-sum
block. Worth keeping near the front of the mental cache.

### 5. Sorting unlocks "closest pair" (CodeChef HORSES)
"Min |a − b| over all pairs in an array" is **O(n log n)** with sort
+ adjacency scan, not O(n²). The proof is one line — for any three
sorted values a < b < c, both b−a and c−b are ≤ c−a, so the optimum
is always between adjacent elements — but the takeaway is bigger:

> Sorting often turns a 2D ("any pair") question into a 1D
> ("adjacent pair") question.

Same idea will show up later in: meeting-room overlaps, "minimum
moves to make array equal", and the closest-points-on-a-line
problem.

### 6. Simulation with running invariants (CodeChef TLG)
TLG is a clean reminder that not every problem is a clever trick.
Sometimes you just **carry forward the right invariants** —
running totals, current max — and the answer falls out. The skill
is identifying *which* quantities to carry: total1, total2, and the
best lead so far. Once those are pinned down, the loop body is
mechanical.
