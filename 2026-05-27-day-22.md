## Today's work

### LeetCode (1)

1. **Coin Change II (#518, Medium)** — given an `amount` and a list
   of distinct coin denominations, return the *number of
   combinations* (multisets) that sum to `amount`. Each coin may be
   used any number of times.
   - **Recurrence:** `dp[a] += dp[a - c]` for every coin `c <= a`,
     with `dp[0] = 1` (the empty multiset is the one way to make
     zero).
   - **Loop order is the lesson:** outer = coins, inner = amount.
     Swapping them counts permutations instead of combinations.
   - **Why outer-coins counts combinations:** when we process coin
     `c`, every dp value we update is built on top of dp values that
     already accounted for *only* coins `<= c` in the iteration
     order. So a multiset like `{1, 2, 2}` is counted exactly once —
     when we "extend the way of making 3 using {1, 2}" by an
     additional 2 in the coin=2 pass. Order of placement never
     enters the count.
   - **Why outer-amount counts permutations:** if instead we fix
     `a` and loop over all coins, the same multiset `{1, 2, 2}` is
     reached three times: as `1+2+2`, `2+1+2`, and `2+2+1`. That is
     the right shape for problems like **#377 Combination Sum IV**
     (despite the misleading name — it actually counts permutations)
     and **#70 Climbing Stairs** (where each step is an ordered
     event).
   - **The +0 base case:** `dp[0] = 1`. This is the empty-product /
     empty-sum convention: there is exactly one way to make zero
     (use nothing). Forgetting this is the #1 bug — without it
     every `dp[a]` stays 0 forever because there is no seed to
     propagate.
   - **Complexity:** Time O(amount × len(coins)), space O(amount).
   - Verified across 13 cases including the LC samples
     (`amount=5, coins=[1,2,5]` → 4), parity-unreachable
     (`amount=7, coins=[2,4]` → 0), the 32-bit stress
     (`amount=500, coins=[1,2,5]` → 12701), and the empty-coins
     edges (`amount=0, coins=[]` → 1).

### CodeChef (1)

1. **SUPW (ZCO14002, Easy-Medium)** — Calvin has `N` consecutive
   days. Day `i` carries cost `a[i]` minutes. Calvin must "do
   SUPW" on some subset of days such that no 3 consecutive days are
   skipped, minimising total minutes.
   - **Recurrence:** `dp[i] = a[i] + min(dp[i-1], dp[i-2], dp[i-3])`
     with the convention that `dp[i-r]` is `0` (treated as out of
     range) when `i-r < 0`. Equivalently: `dp[0]=a[0], dp[1]=a[1],
     dp[2]=a[2]` and the loop starts at `i=3`.
   - **`dp[i]` is the min cost when day `i` is the last selected
     day** — not the min cost on the prefix `[0..i]`. That subtle
     framing is what makes the answer **`min(dp[N-3], dp[N-2],
     dp[N-1])`** instead of just `dp[N-1]`: the last selected day
     must lie in the last 3 positions (otherwise the final window
     of 3 days has no SUPW pick, violating the constraint).
   - **Why 3?** "No 3 or more consecutive unselected" means the gap
     between any two selected days is at most 3 positions (skip 2
     between picks). So the previous selected day was either `i-1`,
     `i-2`, or `i-3`. The min over those three transitions is the
     whole story.
   - **Why O(1) space works:** the recurrence only ever looks back
     at the previous three `dp` values. Keep three rolling scalars
     (`d1, d2, d3`) instead of an `O(N)` array.
   - **Edge cases:** `N=1` → `a[0]` (trivially the only pick). `N=2`
     → `min(a[0], a[1])` (either day alone covers the only window).
     `N=3` → `min(a)`. Code handles all three under the single
     `if n <= 3: print(min(a))` branch.
   - **Complexity:** Time O(N), space O(1).

## Key concepts

### 1. Combinations vs. permutations in unbounded knapsack
Today's #518 vs. yesterday's #322 is the cleanest possible drill on
*loop nesting decides what you count*. Same array, same coin
denominations, same `dp` recurrence shape — the *order of the two
loops* decides whether the answer counts combinations (unordered
multisets) or permutations (ordered sequences):

| Loop nesting              | What it counts        | Example problem        |
|---------------------------|-----------------------|------------------------|
| outer coins, inner amount | combinations          | **#518 Coin Change II**|
| outer amount, inner coins | permutations          | **#377 Combination Sum IV**, **#70 Climbing Stairs** |

The trick to remember: in the "outer coins" pass, when we update
`dp[a] += dp[a - c]`, the right-hand `dp[a - c]` already represents
"ways to make `a - c` using only coins that we have processed so
far (i.e., this coin or earlier coins)". So extending it by one
more copy of `c` produces a multiset, never a new ordering. In the
"outer amount" pass, the right-hand `dp[a - c]` represents ways to
make `a - c` using *any* coin, including coins we have not yet
"committed to" — so the same multiset gets counted once per coin
that appears as the last addition.

### 2. The empty-product / empty-sum base case
`dp[0] = 1` for Coin Change II, `dp[0] = 0` for Coin Change I, and
`dp[0] = 1` for nearly every "count of ways" DP (subset-sum count,
LCS count, partition count). The empty case is **one** way to do
*nothing* — the empty multiset, the empty subsequence, the empty
partition. This is a convention shared with the empty sum being 0
and the empty product being 1.

The contrast with min/sum DPs is sharp: in #322 we want **0 coins**
to make amount 0, so `dp[0] = 0`. In #518 we want **1 way** to make
amount 0, so `dp[0] = 1`. Same problem family, opposite base case,
because the *aggregation* changed from "min cost" to "count of
configurations".

### 3. "Last selected element" DPs and their final-window guard
SUPW is the cleanest example of a recurring shape: a DP where
`dp[i]` is the answer *conditional on `i` being the last picked
element*, not the answer for the whole prefix. The final answer is
then a min/max/sum over the last `k` indices, not just `dp[N-1]`.

This shape appears in:

- **Longest Increasing Subsequence (#300)** — `dp[i]` = length of
  LIS ending at index `i`. Final answer is `max(dp)`, not `dp[N-1]`.
- **Maximum Subarray (#53) (Kadane's variant)** — `dp[i]` = max
  subarray sum ending at index `i`. Final answer is `max(dp)`.
- **Jump Game II / Frog Jump** — variants where the last allowed
  jump width forces the answer to be aggregated over the last `k`
  positions.

The mental discipline: whenever a DP definition says "ending at
`i`" or "with `i` as the last X", the final answer is **not**
`dp[N-1]` — it's an aggregation over the suffix.

### 4. Rolling scalars to drop space from O(N) to O(1)
SUPW only looks back 3 positions. Coin Change variants only look
back as many positions as the largest coin. Fibonacci-shape DPs
(#70, #198, #213, #509, #1137) only look back 1 or 2. Whenever the
look-back window is bounded and small, the array can be replaced
with that many rolling scalars.

The implementation pattern is always the same:

```python
d1, d2, d3 = base0, base1, base2
for i in range(3, n):
    cur = a[i] + min(d1, d2, d3)
    d1, d2, d3 = d2, d3, cur
```

Worth the few extra lines on any DP problem where memory is a
concern (or where the interviewer is grilling on follow-ups).

### 5. Greedy is wrong here, again
Yesterday I noted that greedy coin change fails for arbitrary
denominations. SUPW has its own greedy trap: "always skip the
2 most expensive of every 3" or "always pick the cheapest of every
3" are both wrong — they ignore overlap between consecutive
3-windows. The DP's strength is that it *considers all 3
predecessors* and lets the recurrence resolve the overlap.

Counterexample to the "pick cheapest of every 3" greedy:
`a = [1, 100, 100, 1, 100, 100, 1]`. Greedy picks days 0, 3, 6 →
sum 3. That's actually optimal here. But for
`a = [1, 100, 1, 100, 1, 100, 1]`, greedy on windows [0..2],
[3..5], [6..6] picks days 0, 4, 6 → sum 3. DP also picks 3. So
this is one of the cases where they agree. Constructing a clean
counterexample is left as a stretch exercise — it's harder than
the coin-change counterexample because SUPW's structure (linear
3-window) gives greedy more wiggle room. The point stands: the
algorithm has to be the DP, because greedy's correctness depends
on properties of the input we don't get to assume.
