## Key concepts

### 1. "Binary search the answer" is just lower_bound on a predicate
Days 8–9 set up the two templates (closed `[lo, hi]` and half-open
`[lo, hi)`); today reframes both: the "array" doesn't have to be a
sequence in memory — it can be the *integer answer space itself*.
The predicate becomes the question we're asking ("does `r * r` already
exceed `x`?", "is this version bad?", "can we ship in ≤ k days at
rate `mid`?"). Once that mental flip clicks, problems #69, #278,
#367, #875 (Koko), #1011 (ship-packages) all collapse to the same
two lines of code with a different predicate.

Cheat-sheet of predicates we used today:

| Problem | Predicate                | Template return                  |
|---------|--------------------------|----------------------------------|
| #278    | `isBadVersion(mid)`      | `lo` (leftmost True)             |
| #34 lo  | `nums[mid] >= target`    | `lo` (leftmost ≥ target)         |
| #34 hi  | `nums[mid] >  target`    | `lo` (leftmost > target); last = `lo - 1` |
| #69     | `mid * mid >  x`         | `lo - 1` (largest feasible)      |
| #367    | exact `mid * mid == num` | found / not-found (closed style) |

The unifying frame: every monotonic-predicate question has a
"boundary index" between the False region and the True region.
`lower_bound` returns that boundary. Everything else is bookkeeping
around whether the boundary itself, or the slot just before it, is
the answer.

### 2. `bisect_left` and `bisect_right` are `lower_bound` and `upper_bound`
Python's standard library already implements the two boundary
searches, and they're worth memorising as names:

- `bisect.bisect_left(a, t)` -> `lower_bound`: leftmost `i` with
  `a[i] >= t`. The "insert keeping the duplicates on the right."
- `bisect.bisect_right(a, t)` -> `upper_bound`: leftmost `i` with
  `a[i] >  t`. The "insert keeping the duplicates on the left."

Number of occurrences of `t` in a sorted array = `bisect_right(a, t)
- bisect_left(a, t)`. That single identity solves a surprising
number of "how many equals?" / "first index" / "last index"
follow-ups in O(log n).

In a contest where you can use the standard library, you write
`bisect_left` / `bisect_right` and move on. In an interview where
you can't, you write the half-open template from memory — and
having implemented both myself today, the template is now muscle
memory.

### 3. The `mid * mid` overflow trap and the integer-only `isqrt` style
In Python `mid * mid` can grow arbitrarily large with no penalty,
but the *portable* form is `mid > x // mid` (or `mid > x / mid` in
C/C++ when you accept a small precision risk). The reason: in
fixed-width languages, `mid * mid` overflows once `mid` exceeds
`sqrt(INT_MAX) ≈ 46340`. The division form moves one factor of `mid`
to the other side first, so neither side ever exceeds the input
range.

Pair this with Day 9's "use `i * i <= n`, not `i <= sqrt(n)`" for
trial division: the same overflow-vs-floating-point trade-off
appears across number-theoretic loops. Pick *one* integer-only
idiom and stick with it.

### 4. MUFFINS3 — why the maximum is at `A = N // 2 + 1`
This is a tiny number-theory exercise that's worth understanding,
not memorising. For any `A` in `[1, N]`, `N mod A` is in `[0, A − 1]`,
so the largest possible remainder for a *given* `A` is `A − 1`.
That ceiling is achieved exactly when `A > N / 2`, because then
`floor(N / A) = 1` and the remainder is `N − A`. So we want the
smallest `A` greater than `N / 2`, which is `A = N // 2 + 1`. That
choice gives remainder `N − (N // 2 + 1)`, i.e. roughly `N / 2 − 1`
— and the tie-break rule ("largest A among the maximisers") is
already satisfied because no `A` in `(N/2, N]` beats it.

Worked example, `N = 10`: `A ∈ {6, 7, 8, 9, 10}` give remainders
`{4, 3, 2, 1, 0}`. Max is 4 at `A = 6 = 10 // 2 + 1`. ✓

### 5. "Closed-form before brute-force" is a competitive-programming reflex
MUFFINS3 is the kind of problem where the temptation is to loop
over every `A` in `[1, N]` and pick the max. With `N ≤ 10⁸` and
`T ≤ 1000`, that's 10¹¹ operations — TLE in any language. But the
five-line algebra above replaces the loop with a single arithmetic
operation. The competitive-programming reflex to build:

  *"Before I write a loop, can I prove a one-liner?"*

This is also why I included the brute-force vs formula sweep in the
sandbox test: not to ship the brute force, but to *verify the
closed-form against it* on small inputs. Whenever I derive a
formula, brute-force on `N ∈ [1, 50]` is cheap insurance.
