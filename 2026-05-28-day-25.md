## Today's work

### LeetCode (1)

1. **Max Area of Island (#695, Medium)** — given an `m x n`
   binary grid of `0` (water) and `1` (land), return the
   largest island's area. An island is a 4-directionally
   connected set of `1`s; area is the number of cells in it.
   Return `0` if there are no islands.
   - **Same template as #200, different reduction.** The
     outer scan + iterative DFS + in-place visited marking
     is byte-for-byte the same as yesterday's
     `numIslands`. The only diff: instead of
     `count += 1` on first hit, we walk the whole component
     while accumulating `area += 1`, then update
     `best = max(best, area)` at the end of the component.
     Once you've internalised the flood-fill skeleton, all
     of these problems reduce to "what do I accumulate
     during the walk."
   - **Watch the types.** #200 had `grid[r][c] == "1"`
     (strings); #695 has `grid[r][c] == 1` (integers).
     Copy-pasting the #200 solution without changing the
     comparison would silently return `0` on every grid.
     The starter signature
     `def maxAreaOfIsland(self, grid: List[List[int]]) -> int`
     is the only hint LeetCode gives you about this — read
     the type annotation before writing the body.
   - **In-place mutation: same trade-off as before.** Set
     visited cells to `0` to fuse the visited check with
     the "is this water" check. Cost: caller loses the
     original grid. For LeetCode that's fine; in
     production code, copy first.
   - **Iterative DFS, again.** Worst case is a snake island
     filling the whole `50 x 50` grid (2500 cells). Default
     Python recursion limit (1000) would blow up. An
     explicit stack (`deque` for `O(1)` pops) is the
     defensive default — extra five lines of code for
     unbounded depth.
   - **Edge cases worth holding in mind:** empty grid `[]`,
     empty row `[[]]`, all water (`0`), all land (return
     `m * n`), single-cell island, single-cell water,
     diagonal-only land (each cell its own island of size
     1 — not 4-connected through corners), donut with
     interior cell (outer ring is *not* connected to the
     centre — moat in between). All exercised in the
     verification scratch.
   - **Complexity:** Time `O(m * n)` — each cell is pushed
     and popped at most once. Space `O(m * n)` worst case
     for the stack on a pathological snake.

### CodeChef (1)

1. **DIFFSUM — Sum OR Difference (Beginner)** — read two
   integers `N1` and `N2`. Print `N1 - N2` if `N1 > N2`,
   otherwise `N1 + N2`. Constraints: `-1000 <= N1, N2 <=
   1000`.
   - **One line.** `print(a - b if a > b else a + b)` is
     the whole program, and a conditional expression
     (ternary) reads naturally for this kind of
     "two-branch print." The verbose `if/else` form is
     equally fine and arguably easier to debug in a
     contest under pressure — pick whichever your eyes
     parse faster at midnight.
   - **Note: no `T` test-case loop.** Unlike most CodeChef
     beginner problems (which open with
     `t = int(input())` and loop), DIFFSUM is a single
     case — `N1` on line 1, `N2` on line 2. Read the
     "Input" section before reaching for the boilerplate;
     muscle memory will add a `T` loop that isn't there
     and the grader will time out reading nothing.
   - **The `>` (strict) detail.** "If first greater than
     second" means strict — equal values fall into the
     `else` branch and print the sum. With `a = b = 5` you
     get `10`, not `0`. Easy to flip with `>=` and not
     notice on the sample (which uses `82, 28`).
   - **Complexity:** Time `O(1)`. Space `O(1)`.

## Key concepts

### 1. Flood-fill template: one skeleton, many reductions

The grid-flood-fill template established yesterday on
**#200** is the engine for an entire family of grid problems.
The skeleton:

```
for r in range(rows):
    for c in range(cols):
        if grid[r][c] != 1:
            continue
        # ... iterative DFS / BFS that walks the whole
        # connected component, marking visited as it goes ...
        # ... accumulate something per cell, do something
        # with the accumulator at component end ...
```

What changes between problems is **what you accumulate** and
**what you do with the accumulator at the component end**:

| Problem | Per-cell update | Per-component finish |
|---------|-----------------|----------------------|
| #200 Number of Islands | (nothing) | `count += 1` |
| #695 Max Area of Island | `area += 1` | `best = max(best, area)` |
| #463 Island Perimeter | `perim += 4 - land_neighbours` | (sum across components — but only one in #463) |
| #1254 Closed Islands | (nothing) | `count += 1` *only* if the component never touched the border |
| #547 Number of Provinces | (nothing) | `count += 1` — same as #200 but on an adjacency matrix |
| #1905 Count Sub-Islands | track "every cell also in grid2" | `count += 1` if flag stayed true |

Same flood, different bookkeeping. Once the flood-fill
itself is automatic, all of these become five-minute
problems.

### 2. Type annotations are the contract — read them first

LeetCode's starter template for #695 is
`def maxAreaOfIsland(self, grid: List[List[int]]) -> int`.
That `List[List[int]]` (not `List[List[str]]`) is the only
thing distinguishing the grid format from #200's
`List[List[str]]`. A solution copy-pasted from #200 with
`grid[r][c] == "1"` will compile, run, and silently return
`0` for every test case. The lesson: **before writing the
body, glance at the signature**:
- Return type: am I returning the right shape?
- Parameter types: am I comparing to the right value?
- Default values: am I shadowing them?

This is also a routine source of WA (wrong answer) in
contest problems where two near-identical problems use
slightly different input encodings.

### 3. Conditional expressions vs `if/else`

Python's `a if cond else b` is a conditional **expression**
— it has a value, so you can use it inline. The verbose
`if/else` form is a **statement** that controls flow but
returns nothing.

```python
# expression — usable as a value
print(a - b if a > b else a + b)

# statement — same effect, more lines
if a > b:
    print(a - b)
else:
    print(a + b)
```

For one-liners like DIFFSUM the expression form is cleaner.
For multi-branch logic or anything with side effects
(logging, error handling, multiple updates), the statement
form scales better. **Rule of thumb:** if the expression
form fits comfortably on one line and has no side effects,
use it; otherwise reach for the statement form.

### 4. Strict vs non-strict comparison

A bug class that shows up everywhere from sorting to
greedy algorithms to game-theory invariants: **strict (`>`,
`<`) vs non-strict (`>=`, `<=`) comparison**. They differ
*only* on equality, which is exactly where the bug hides
because most random tests won't generate ties.

In DIFFSUM, the spec says "if the first number is greater"
— that's strict. `a == b` goes to the `else` branch and
prints the sum. A solution using `a >= b` would print
`a - b = 0` for equal inputs instead of `a + b = 2a`. The
sample input `(82, 28)` doesn't catch this. **Always think
through what equal inputs should do**, then pick the
operator that matches.
