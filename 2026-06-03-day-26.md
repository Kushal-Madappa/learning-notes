## Key concepts

### 1. DFS over three data structures — the connected-components engine

The same connected-components-count engine has now been written three
times this week against three different graph representations. The
skeleton:

```
for each node:
    if not visited:
        components += 1
        iterative DFS marking everything reachable as visited
```

The only thing that changes per problem is **how you enumerate `u`'s
neighbours**:

| Representation              | "Neighbours of u" code                                  | Problem |
|-----------------------------|---------------------------------------------------------|---------|
| 2-D grid (4-connected)      | for `(dr, dc)` in `(-1,0), (1,0), (0,-1), (0,1)`        | #200, #695 |
| Adjacency matrix `n x n`    | `for v in range(n) if isConnected[u][v]`                | #547    |
| Adjacency list `adj[u]`     | `for v in adj[u]`                                       | #1971   |
| Edge list (preprocess once) | build `adj` first, then same as adjacency list          | most graph problems with `edges: List[List[int]]` |

The grid problems and the matrix problem all run in O(n^2) because
they have to look at every cell of the input. The adjacency-list
problems run in O(V + E), which is strictly better whenever `E ≪ V^2`
(sparse graphs — most real graphs). The decision rule:

- **n ≤ 200 (LeetCode small-graph problems):** adjacency matrix is
  fine; the O(n^2) factor is at most 4 × 10^4 operations.
- **n ≤ 10^5 (CP large-graph problems):** adjacency list is the only
  feasible representation; allocating an `n^2` matrix at `n = 10^5`
  is already 10^10 cells and won't fit in RAM.

### 2. Why the iterative pattern matters even when recursion fits

LeetCode caps `n` at 200 for #547. Recursive DFS would survive on
Python's default 1000-frame stack. **Iterative is still the right
default** because:

- **Portability.** The exact same code works at n = 10^6 with no
  `sys.setrecursionlimit` hack and no risk of segfault from a deep
  C-stack underneath the Python frame.
- **Constant factor.** A `while stack` loop with `deque.pop()` is
  meaningfully faster than recursive Python calls (saves ~1 µs per
  call). At `n = 200` the difference is invisible, but the muscle
  memory transfers.
- **Debuggability.** Iterative DFS can be paused, the stack inspected,
  and the visit order verified by printing the stack contents. The
  recursive version requires `pdb` to peek at frames.

The only time recursion is genuinely cleaner is when the *result* of
the recursion is the value you want (tree DP problems — return the
subtree's answer). Connected-components doesn't fit that shape —
the DFS only mutates `visited`, it doesn't return a value per node.

### 3. Adjacency matrix vs. adjacency list — the memory trade

| Property               | Adjacency Matrix | Adjacency List |
|------------------------|------------------|-----------------|
| Space                  | `O(V^2)`         | `O(V + E)`      |
| "Is u adjacent to v?"  | `O(1)`           | `O(deg(u))`     |
| Iterate u's neighbours | `O(V)`           | `O(deg(u))`     |
| Add an edge            | `O(1)`           | `O(1)`          |
| Remove an edge         | `O(1)`           | `O(deg(u))`     |

The matrix wins on **dense graphs** (E close to V^2) and on **edge
lookup**. The list wins on **sparse graphs**, which is most real
graphs, and on **neighbour iteration**, which is what every traversal
needs. CodeChef and LeetCode problems usually have V ≤ 200 (matrix
is fine) or V ≤ 10^5 with E ≤ 2 × 10^5 (list is the only choice).

### 4. The connected-components reduction zoo

Once you've written the skeleton three times, you start noticing
that the **per-component computation** is the only variable:

| Goal                                  | Per-component bookkeeping        | Final answer        |
|---------------------------------------|----------------------------------|---------------------|
| Count components (#547, #200)         | none — just increment            | total count         |
| Largest component area (#695)         | `area += 1` during the DFS       | `max(best, area)`   |
| Component sizes list (e.g., later)    | append size after each DFS       | the list itself     |
| Component sum / weight                | `sum += value[u]`                | aggregate of sums   |
| Check if u and v in same component (#1971) | track which component each node is in | `comp[u] == comp[v]`? |
| Topological order (DAG, later)        | append `u` after recursive return| reversed list       |
| Bipartite check (2-colouring, later)  | assign alternating colors        | did any conflict?   |

The DFS skeleton is the same. The reduction inside the DFS changes
based on what you want. This is the same pattern as tree DP —
recurse, then aggregate — but adapted to graphs that may have cycles.
