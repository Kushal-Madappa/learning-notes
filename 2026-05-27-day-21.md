## Today's work

### LeetCode (1)

1. **Coin Change (#322, Medium)** — given coin denominations and a
   target `amount`, return the *fewest* coins that sum to `amount`,
   or `-1` if impossible.
   - **Recurrence:** `dp[a] = min(dp[a - c] + 1)` over every coin `c
     <= a`, with `dp[0] = 0` and a sentinel `dp[a] = amount + 1` for
     "unreachable so far".
   - **Why bottom-up beats top-down here:** the dependency order
     `a = 1, 2, …, amount` is trivial, the state space is small
     (`amount <= 10^4` in LeetCode constraints), and the table is
     densely visited (every `a` from 1 to `amount` is computed
     exactly once). Top-down would add Python's function-call
     overhead for no upside.
   - **Why it is *unbounded* knapsack:** each coin can be reused
     freely. The reuse is implicit because `dp[a - c]` may already
     have used coin `c` itself — no "don't pick i again" bookkeeping.
     Contrast with 0/1 knapsack (#416 Partition Equal Subset Sum),
     where each item is used at most once and the outer loop has to
     iterate items (with the inner loop going *backwards* over the
     target to prevent reuse).
   - **Loop order subtlety (write this down):**
     - **Min-coins-to-make-amount (#322):** loop order doesn't matter
       because we're taking a `min`, and `min` is commutative across
       coin orderings.
     - **Number of ways to make amount (#518 Coin Change II):** loop
       order *does* matter. Outer = coins, inner = amount counts
       *combinations*; outer = amount, inner = coins counts
       *permutations*. The user almost always wants combinations,
       so for #518 the outer is coins.
   - **The "+1 sentinel" trick:** initialising `dp` with `amount + 1`
     instead of `float('inf')` keeps everything in C-int arithmetic,
     avoids float comparisons, and cleanly marks unreachable states
     (any real answer is at most `amount` using all 1-coins if
     present, so `amount + 1` is a safe "impossible" marker).
   - **Edge cases that catch beginners:** `amount = 0` returns `0`
     (zero coins make zero); `coins = [2]`, `amount = 3` returns `-1`
     (parity-unreachable); `coins = [1, 2147483647]`, `amount = 2`
     returns `2` (a huge denomination doesn't help if it's still
     larger than `amount`).
   - **Complexity:** Time O(amount × len(coins)), space O(amount).
   - Verified across 12 cases including the classic LC trickies
     (`[186, 419, 83, 408]`, `amount = 6249` → 20) and the parity-
     unreachable edge (`[2, 4]`, `amount = 7` → -1).

## Key concepts

### 1. Unbounded vs. 0/1 knapsack — the loop-order discipline
Today's #322 is the first *unbounded* knapsack problem of this run.
The difference from 0/1 knapsack is one line of code, but the
intuition cuts deep:

| Variant            | Outer loop  | Inner loop direction | Why                                |
|--------------------|-------------|----------------------|------------------------------------|
| Unbounded (#322)   | target `a`  | over coins (any dir) | coin can be reused, so `dp[a - c]` may already contain coin `c` |
| 0/1 (#416, #494)   | items       | target *backwards*   | each item used at most once; reverse iteration prevents reuse within the same `i` |
| Count of ways (#518)| coins (outer), target (inner) | forwards | enforces combination (not permutation) ordering |

The decision tree at runtime is the same — *take this coin or skip
it* — but the *order in which states are populated* dictates whether
"take" means "take once" or "take any number of times". This is the
core invariant to internalise before #494 Target Sum and
#1049 Last Stone Weight II later in the DP wave.

### 2. The "+1 sentinel" pattern
A recurring idiom in min-cost DPs: pick a value strictly greater than
any possible real answer, use it as `INF`, then check `if dp[goal] == INF
→ return -1 (or unreachable)`. For #322, `amount + 1` works because
any real answer is at most `amount` (using all 1-coins if available)
and at least 0. The same idiom shows up in:

- **#787 Cheapest Flights Within K Stops**: `INF = float('inf')` or
  any value greater than `max edge weight * K`.
- **#62-style grid DPs**: when "no path" exists, mark with a sentinel
  greater than any achievable cost.
- **Subset-sum impossibility (e.g. #416)**: use a boolean DP, but
  the *value* variants (#494) need the +1 / +INF dance.

The win over `float('inf')` is staying in integer arithmetic and
avoiding the rare numerical oddity (`float('inf') + 1 == float('inf')`
is fine, but mixing into comparisons with very large ints in some
languages is messier).

### 3. Greedy coin change vs. DP coin change
FLOW005 uses denominations `(1, 2, 5, 10, 50, 100)`. Greedy works
because this system is **canonical** — for every amount, the greedy
choice (always take the largest coin that fits) is optimal. The
formal property: a coin system is canonical iff greedy equals DP for
every amount up to `2 * max(coin)`.

LC #322 uses *arbitrary* denominations. The classic counterexample
is `coins = [1, 3, 4]`, `amount = 6`: greedy takes 4 + 1 + 1 = 3
coins, but DP finds 3 + 3 = 2 coins. So the moment a problem allows
arbitrary coin sets, greedy is wrong and DP is the answer.

**Rule of thumb:** if the coin system is the canonical Indian /
European / US currency, greedy is fine. If the coins are arbitrary
(or chosen adversarially), reach for DP.

### 4. Set membership is the antidote to substring matching
FRGTNLNG is a recurring trap dressed up as a string problem. Beginners
do:

```python
if forgotten_word in modern_line:   # WRONG
```

This is *substring* matching — `"pi" in "piygu"` is `True`, but
`piygu` and `pi` are different words. The fix is to split the
modern line and check set membership:

```python
modern_words = set(modern_line.split())
if forgotten_word in modern_words:  # RIGHT
```

This same trap appears anywhere a "word" / "token" / "identifier" is
compared inside a longer text: log parsing, csv column lookups, JSON
field matching with stringly-typed paths.

### 5. `heapq.nsmallest` vs. `sorted`
SMPAIR asks for the two smallest. The reflex is `sorted(a)[:2]` —
O(n log n). The better answer is `heapq.nsmallest(2, a)` — O(n) with
a bounded min-heap of size `k`. For `k ≪ n` (which is the typical
case for "top-k" / "bottom-k" problems), the difference is
asymptotically large and shows up in CP-style hot loops.

The same idea generalises to `heapq.nlargest`, the running median
two-heaps pattern (#295), and the Kth-element family (#215, #703,
#973). Whenever a problem asks for "the k smallest / largest", the
heap-based answer is usually right.

### 6. Self-inverse permutations
PERMUT2 introduces a small but elegant idea: a permutation `p` is
its own inverse iff `p ∘ p == identity`. Equivalently, the
permutation's cycle decomposition consists only of fixed points and
2-cycles (swaps).

This gives three ways to test:

1. **Direct application:** check `p[p[i]-1] == i+1` for every `i`.
   O(n) time, O(1) extra space. Today's choice.
2. **Compute the inverse:** build array `inv` where `inv[p[i]-1] = i+1`,
   then check `inv == p`. O(n) time, O(n) extra space.
3. **Cycle analysis:** every cycle has length 1 or 2. Useful when
   you want *more* than yes/no — e.g., count the number of fixed
   points vs. swaps.

The 2-cycle / fixed-point structure also shows up in:

- **Bijection problems** where a function is its own inverse.
- **Sorting by swaps** — the minimum number of swaps to sort a
  permutation equals `n - (number of cycles)`.
- **Graph problems** where edges are self-inverse (undirected
  graphs are exactly this).
