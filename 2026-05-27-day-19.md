## Today's work

- **LeetCode (1):**
  - **Climbing Stairs (#70, Easy)** — count the distinct ways to
    reach step `n` when each move is +1 or +2. The recurrence
    `f(n) = f(n-1) + f(n-2)` falls out of "the last move is either a
    1-step from `n-1` or a 2-step from `n-2`," and once you write it
    you've recognised that the answer sequence is Fibonacci with
    `f(1)=1, f(2)=2`.
    - **What I shipped:** bottom-up two-variable rolling DP. Time
      O(n), space O(1). The constraint `n <= 45` means even a naive
      `int` recurrence is comfortable — no big-integer issues, no
      modular arithmetic.
    - **The trap I deliberately avoided:** writing it as recursion
      first. Plain `climb(n) = climb(n-1) + climb(n-2)` is O(2^n) —
      the recursion tree is the Fibonacci tree, which has roughly
      `phi^n` nodes (≈ 1.6 × 10^9 for n=45). Memoisation rescues it
      to O(n), but the bottom-up rewrite is the same O(n) with zero
      stack and zero dict overhead. **For any 1-D DP where the
      dependency order is obvious, default to bottom-up.**
    - **The "rolling array" insight:** `f(i)` only ever looks back
      two steps, so storing the whole DP table is wasteful. Drop the
      array, keep two scalars. This same trick reappears in every
      Fibonacci-shaped DP (#509 Fibonacci Number, #1137 N-th Tribonacci,
      #746 Min Cost Climbing Stairs).
    - **Complexity:** O(n) time, O(1) space.
    - Verified: 10 sequential cases (n=1..10 against the
      Fibonacci-shift table), two LC-provided examples, and both
      boundary values n=44 and n=45. All 14 pass.

- **CodeChef (1):**
  - **Paying Up (MARCHA1, Easy)** — given `n <= 20` banknotes and a
    target `m`, decide whether some subset of the notes sums to
    exactly `m`. Output `Yes` / `No` per test case.
    - **What I shipped:** bitmask brute force. Iterate `mask` from
      `0` to `2^n - 1`; for each, sum the notes whose bit is set in
      the mask; short-circuit on the first match.
    - **Why brute force is the right tool here:** `n <= 20` gives
      at most `2^20 ≈ 10^6` subsets, each scanned in O(n) for the
      sum. That's ~`2 × 10^7` operations per test case — comfortable
      for the typical CodeChef limits (the problem allows ~100 test
      cases, so ~`2 × 10^9` worst case in C, but in practice many
      masks short-circuit and Python handles it fine on the actual
      test data).
    - **Why the bitmask encoding beats recursion in Python:**
      Recursion gives the cleaner "include / exclude" structure that
      every editorial uses, but each recursive call costs a Python
      stack frame (~1 µs). At 2^20 calls that's seconds. The bitmask
      loop is one tight Python `for` over integers — about 10× faster
      in practice.
    - **The connection to next week's DP:** This problem screams
      *subset-sum DP* — the textbook O(n × m) tabulation with
      `dp[i][s] = dp[i-1][s] OR dp[i-1][s - a[i]]`. But `n <= 20`
      makes brute force strictly faster than DP unless `m` is small
      (which the problem doesn't guarantee). The DP version is what
      shows up at LC #416 / #494, where `n` grows to 200+ and brute
      force is dead. **Lesson:** match the algorithm to the
      constraints, not to the elegance.
    - **Complexity:** O(2^n × n) time per test case, O(n) space for
      the input list.
    - Verified: canonical 3-case sample (`[1,2,4]` target 4 → Yes,
      `[2,3,7,8]` target 5 → Yes, `[1,2,4]` target 8 → No) + 6
      hand-built edges (single-note match, single-note miss, full-sum
      target, full-sum-plus-one impossible, n=20 max-size full sum,
      greedy-fails-must-skip-largest). All 9 pass.

## Key concepts

### 1. The DP recipe in its smallest form
Climbing Stairs is the prototype for every 1-D DP that follows. The
five-step recipe from yesterday (SUMTRIAN) applies verbatim:

1. **State.** `f(i)` = number of ways to reach step `i`. One
   dimension, no extra parameters.
2. **Transition.** `f(i) = f(i-1) + f(i-2)`.
3. **Base case.** `f(1) = 1`, `f(2) = 2`.
4. **Overlap check.** `f(5)` is asked for by both `f(7)` and `f(6)` in
   a top-down recursion; the overlap grows exponentially. Memoize, or
   tabulate.
5. **Top-down vs. bottom-up.** Dependency order is `i = 3, 4, …, n`.
   Trivially obvious → bottom-up wins, and the rolling-window
   compresses the table to O(1) space.

This is the template to internalise before #198 House Robber tomorrow,
where the transition becomes `f(i) = max(f(i-1), f(i-2) + nums[i])` —
same shape, one `max` and one input vector heavier.

### 2. The brute-force baseline before DP
MARCHA1 is the canonical "DP problem you don't have to solve with DP
*yet*." With `n <= 20`, the subset-enumeration brute force is
2^n × O(n) ≈ 2 × 10^7 — comfortable. With `n = 200`, the same brute
force is 1.6 × 10^62 operations and you have to switch to the
O(n × m) tabulation:

```
dp[i][s] = can we hit sum s using a prefix of the first i notes?
dp[0][0] = True; dp[0][s>0] = False
dp[i][s] = dp[i-1][s]                      # skip note i
        or (s >= a[i] and dp[i-1][s - a[i]])# take note i
```

The two algorithms have very different complexities and the *constraint
on n* tells you which to reach for. This is a recurring decision in CP:

| n         | Right tool                                    |
|-----------|-----------------------------------------------|
| n <= 20   | Bitmask enumeration (2^n)                     |
| n <= 40   | Meet-in-the-middle (2 * 2^(n/2))              |
| n <= 200  | O(n * target) subset-sum DP                   |
| n large + small target | bitset-accelerated DP (1 + target / 64) |

**Memorise the 20 → bitmask, 40 → MITM, 200 → DP table.** Those
thresholds are the most common CP signposts for subset-sum-shaped
problems.

### 3. Rolling arrays and the "two-scalar" compression
Climbing Stairs needs only `f(i-1)` and `f(i-2)` to compute `f(i)`.
Storing the whole table is O(n) memory for no benefit — the rolling
two-variable form keeps the same time complexity at O(1) space. The
same idea generalises:

- **Two-step lookback** (Fibonacci-shape): keep `prev`, `curr`.
  Examples: #70, #509, #198, #746.
- **k-step lookback**: keep a deque of the last `k` values.
  Examples: #1137 Tribonacci (k=3).
- **Row-only lookback** in 2-D DP: keep the previous row instead of
  the full grid. Examples: #62 Unique Paths, #1143 LCS — drop the
  table from O(m * n) to O(min(m, n)).

The rule: **before allocating the full DP table, check what the
recurrence actually reads from. Often you only need the last row /
last two values.** This is where most DP problems become
production-grade.

### 4. The "include / exclude" decision tree — DP's underlying shape
Both today's problems are decision trees in disguise:

- Climbing Stairs: at each step `i`, the choice is *1-step or 2-step*.
  Total leaves = paths. The recurrence `f(i) = f(i-1) + f(i-2)`
  collapses identical sub-trees rooted at the same `i`.
- MARCHA1: at each banknote, the choice is *include or exclude*.
  Total leaves = 2^n. Brute force enumerates them; subset-sum DP
  collapses identical sub-trees rooted at the same `(i, remaining)`.

This shape is everywhere in DP: every "binary decision per item"
problem has the same skeleton. Once you see it, you can write the DP
in two passes — first enumerate the decision tree, then identify
which `(state)` tuples cause re-visits.
