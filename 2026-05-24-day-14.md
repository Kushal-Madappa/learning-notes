## Today's work

- **LeetCode (1):**
  - **Same Tree (#100, Easy)** — the second tree-recursion problem,
    and the first one that recurses on *two* trees simultaneously.
    - **Recurrence (parallel descent):**
      ```
      same(None, None)  = True
      same(None, _)     = False
      same(_,    None)  = False
      same(p,    q)     = p.val == q.val
                          AND same(p.left,  q.left)
                          AND same(p.right, q.right)
      ```
    - **Why the base-case ordering matters:** if we test
      `p.val != q.val` before the `None` checks, we crash on
      `AttributeError` the first time one side is None. The
      "both-None → one-None → values" order is the safe
      short-circuit — and it's the same defensive ordering you'll
      reuse in #101 (Symmetric Tree).
    - **Complexity:** O(n) time (n = nodes visited, bounded by the
      smaller tree when they differ), O(h) recursion stack.
    - Verified: 3 official LC samples + 7 hand-built edges
      (both-empty, single-vs-empty, deep-leaf mismatch, deep
      identical 13-node tree, ...). All 10 pass.

- **CodeChef (1):**
  - **Chef and Two Strings (CHEFSTLT, Beginner / 1-star)** — given
    two equal-length strings of lowercase letters and `?`s
    (wildcards), report the **minimum** and **maximum** possible
    Hamming distance after every `?` is replaced with some letter.
    - **Insight: each position is independent.** This is the key
      observation. Because we get to pick each `?`'s replacement
      freely, no position's choice constrains any other's, so the
      minimum and maximum are just the sums of per-position
      minimum and maximum contributions.
    - **Per-position table** (the entire problem fits in one
      truth table):
      ```
                |  same letter  |  diff letters  |  one '?'  |  both '?'
        min  +=     0                 1               0           0
        max  +=     0                 1               1           1
      ```
    - **Reading the table:** "min = mismatched letter pairs only"
      and "max = mismatched letter pairs + every position
      containing at least one '?'". So the code is just two
      counters running through `zip(s1, s2)`.
    - **Algorithm (8 lines):** loop, classify each position,
      increment. O(N) per test. Trivially within bounds (N ≤ 100).
    - Verified: 3 official samples + 5 edge cases (single char
      equal/different, all-'?', identical-no-'?',
      completely-different-no-'?'). All pass.

## Key concepts

### 1. Parallel recursion: the "two-tree template"
The shape #100 installs — and it's the one #101 (Symmetric Tree),
#572 (Subtree of Another Tree), and #1325 (Delete Leaves with a
Given Value, two-tree variants) all reuse — is just yesterday's
template doubled:

```python
def f(p, q):
    if p is None and q is None: return BASE_BOTH_NONE
    if p is None or  q is None: return BASE_ONE_NONE
    if not LOCAL_OK(p, q):      return False
    return f(p.left,  q.left) and f(p.right, q.right)   # parallel
```

Two knobs change between problems:
1. **`LOCAL_OK`** — the per-node consistency check. For #100 it's
   `p.val == q.val`. For #572 ("Subtree") you wrap an *outer*
   recursion that asks "does the subtree at this node match the
   target?" using #100 as a helper. For #1325 it can be a `None`
   check on filtered children.
2. **The recursion order** — for #100 it's `(left, left)` and
   `(right, right)`. For #101 it's the *cross-pairing*
   `(left, right)` and `(right, left)`. That single change is the
   only thing that turns "same tree" into "mirror tree" — file
   that away, it's a common interview prompt.

### 2. The base-case ordering trap (and the safe pattern)
A natural-but-buggy version of #100 looks like this:

```python
def isSameTree(p, q):
    if p.val != q.val: return False     # <-- AttributeError when p is None
    return isSameTree(p.left, q.left) and isSameTree(p.right, q.right)
```

The fix isn't subtle, but the *order* matters: test
`both-None → one-None → values` so that by the time you read
`p.val` and `q.val`, you've already guaranteed both are non-None.
This same order shows up in every two-pointer / two-list problem
(merge two sorted lists, intersection of two linked lists, etc.) —
"both ended → one ended → compare values" is a portable defensive
pattern.

### 3. CHEFSTLT: "each position is independent" → split min/max
When a problem asks for both the **min** and the **max** of some
quantity, and the choices are independent across positions, the
fast move is almost always:

> *write a per-position contribution table, then sum.*

You don't need to think about "what if I picked X here and Y
there" — independence means you can solve each cell on its own.
This generalizes immediately to:
- Range-replace problems where each cell has a fixed set of choices.
- DP-with-no-dependencies problems (often called "greedy" because
  the DP collapses to a sum).
- Many "min cost vs max profit" problems where the same code
  computes both — just swap `min`/`max` in the per-cell rule.

The mental discipline is: **first ask "are the choices coupled?"**
If yes, you need real DP. If no, sum a table. CHEFSTLT is firmly
in the "no" camp; the `?` at position 5 has nothing to say about
the `?` at position 7.

### 4. The "wildcard absorbs everything" rule
Across many string problems with wildcards (`?`, `*`, etc.) the
single most common simplification is: **once at least one side is
a wildcard, the answer at that cell is whatever you want it to
be**. For Hamming-distance variants that means "if either is `?`,
this position contributes 0 to min, 1 to max." For longest-common-
subsequence variants it means "a `?` matches anything." For
regex-style problems (`*` in #44 Wildcard Matching) it means an
entire prefix can be absorbed. Spotting "this cell is a wildcard,
so I get to choose" is the move that collapses three of the
trickier-looking string problems into one-liners.
