## Today's work

### LeetCode (6)

1. **House Robber (#198, Medium)** — first DP with an explicit
   take/skip decision. State `f(i)` = best total robbed from
   `nums[0..i]`. Transition `f(i) = max(f(i-1), f(i-2) + nums[i])`
   reads as *"skip house i"* vs *"rob house i (forces skipping i-1)"*.
   Two scalars suffice — O(n) / O(1). The recurrence has the same
   *shape* as Climbing Stairs (look two back), but now the choice is
   numerical rather than combinatorial.

2. **Min Cost Climbing Stairs (#746, Easy)** — variant of #70 where
   each step `i` carries a cost paid on *leaving* it, and the goal
   is to reach the "top" (index `n`) cheaply. State `f(i)` = min
   cost to *stand on* step i. Transition
   `f(i) = cost[i] + min(f(i-1), f(i-2))`. Answer is `min(f(n-1), f(n-2))`
   because the top is reachable from either of the last two steps.
   The trap is the answer: many beginners return `f(n-1)` and miss
   that step `n-2` is also a launching pad for the top.

3. **Fibonacci Number (#509, Easy)** — the literal Fibonacci, picked
   as the warm-up that confirms the rolling-window reflex. Different
   base case from #70 (`f(0)=0, f(1)=1` vs `f(1)=1, f(2)=2`) but
   identical loop body. Tested against a hardcoded table of `f(0)..f(20)`
   plus the upper bound `f(30) = 832040`.

4. **N-th Tribonacci Number (#1137, Easy)** — extends the rolling
   window to three scalars. Transition `T(n) = T(n-1) + T(n-2) + T(n-3)`,
   bases `T(0)=0, T(1)=T(2)=1`. The interesting bit is the *parallel
   assignment* `a, b, c = b, c, a + b + c` — one Python statement
   per step, no deque. Verified the upper bound `T(37) = 2_082_876_103`,
   which is the exact LC constraint.

5. **Maximum Subarray (#53, Medium)** — *Kadane's algorithm*, the
   first DP today whose state is **best subarray sum ENDING at
   index i** rather than "best up to i". The shift in framing is
   the whole insight: the transition `curr = max(nums[i], curr + nums[i])`
   reads as *"start fresh at i, or extend the previous run"*. One
   scalar for `curr`, one for the global `best`. Handles all-negative
   arrays correctly because the `max(nums[i], ...)` lets `curr`
   reset to a single (possibly negative) element. **The all-negative
   case is the trap** — a naive "ignore negatives" Kadane would
   return `0` and fail.

6. **House Robber II (#213, Medium)** — circular street: house `n-1`
   is now adjacent to house `0`, so they can't both be robbed.
   Clever reduction: solve the linear #198 problem TWICE on the two
   slices that each drop one endpoint (`nums[:-1]` and `nums[1:]`),
   then take the max. Each linear pass is O(n)/O(1); two passes
   = still O(n)/O(1). The reduction trick is the takeaway — many
   "circular" DP variants collapse to "solve the linear version on
   two slices and compare."

### CodeChef (1)

7. **Bytelandian Gold Coins (COINS, Easy)** — given a coin of value
   `n` (up to 10^9), you can either sell it for `n` dollars or
   exchange it into three coins valued `n//2, n//3, n//4` (rounded
   down). Maximise the dollars. Recurrence
   `f(n) = max(n, f(n//2) + f(n//3) + f(n//4))`, base `f(n) = n`
   for `n < 12` (because `n//2 + n//3 + n//4 <= n` once n is small
   enough — for `n=11` the split gives `5+3+2 = 10 < 11`).
   - **Why top-down (memoization) beats bottom-up here:** the
     bottom-up table would need `O(n) = 10^9` entries, which is
     infeasible. But the recursion only ever asks about values of
     the form `n` divided by repeated 2s, 3s, and 4s — a tiny
     fraction of the integers below `n`. `lru_cache` on the recursive
     function visits maybe ~10^4 distinct states for `n = 10^9`.
   - **Why brute recursion without memo fails:** the recursion tree
     branches 3 ways at each level, so naively it's `3^depth` calls.
     For `n = 10^9`, depth is `log_2(n) ≈ 30`, so `3^30 ≈ 2 × 10^14`
     calls. Memoization is the entire game.
   - Verified for `n = 10^9` → 4_243_218_150 (the canonical answer
     cited in every editorial) and confirmed the 2-line CodeChef
     sample (`12 → 13`, `2 → 2`).

## Key concepts

### 1. The "minimum state" question
Every problem today reduces from a textbook O(n)-space DP table to an
O(1)-space rolling window because the recurrence only looks back a
fixed number of steps. The discipline is to **look at the transition
first, then count the indices it references**:

- Looks back 2 steps (#70, #198, #509, #746): keep `prev2, prev1`.
- Looks back 3 steps (#1137): keep `a, b, c`.
- Looks back 1 step plus a global aggregate (#53): keep `curr, best`.
- Looks back 2 steps but problem is circular (#213): solve linearly
  *twice* on two slices.

This compression is not a micro-optimization — it's the difference
between O(n) memory and O(1) memory, and on hot paths it's also a
constant-factor win because the scalars sit in registers.

### 2. Top-down memoization vs. bottom-up tabulation
COINS is the first problem this month where **bottom-up is wrong**.
The state space (`n` up to 10^9) is too big to allocate, but the
reachable states are sparse. The decision rule:

| Situation                                                  | Use        |
|------------------------------------------------------------|------------|
| State space is small and densely visited                   | Bottom-up  |
| State space is small but dependency order is obscure       | Top-down   |
| State space is HUGE but reachable states are SPARSE        | Top-down   |
| Need to compose with iterative compression (rolling array) | Bottom-up  |

This is also why `functools.lru_cache` is the right Python tool for
COINS — one decorator turns an exponential recursion into a near-linear
one with no manual dict bookkeeping.

### 3. The "what does the state index?" pivot — Kadane's reframing
For most 1-D DP I've written, the state is "best answer using prefix
`[0..i]`" — a *global* aggregate. Kadane's flips it: the state is
"best answer of a contiguous subarray ENDING at i" — a *local*
aggregate. The global answer is then `max(f(0), f(1), …, f(n-1))`.

This pivot is essential for problems where the answer must be a
*contiguous* run (subarray, substring) rather than a *selectable*
subset. It generalises to:

- **#152 Maximum Product Subarray** — Kadane with *two* rolling
  scalars (best ending here, worst ending here) because a negative
  can flip rankings.
- **#918 Maximum Sum Circular Subarray** — Kadane twice (regular
  max + min), with a complement trick analogous to the #213 split.
- **#560 Subarray Sum Equals K** — "ending at i" framing combined
  with prefix sums + hash map.

### 4. Circular → two-linear reductions
House Robber II shows the cleanest example of a recurring CP /
interview pattern: when the constraint is *circular*, solve the
*linear* version on two windows that each break the circularity at
a different point, then combine.

Where else this trick appears:

- Maximum Sum Circular Subarray (#918): max of (Kadane on the array)
  vs (`total - minKadane on the array`, excluding the all-negative
  edge case).
- Rotating-array problems where you concatenate `arr + arr` and
  cap the window at length `n`.
- Round-robin scheduling problems where you fix one element's
  position and reduce to a linear chain.

### 5. The COINS breakpoint: when does the split stop helping?
Splitting a coin of value `n` into `n//2 + n//3 + n//4` yields at most
`13n/12` for sufficiently large `n` and `< n` for very small `n`. The
exact crossover: for `n < 12`, the split is at most `5+3+2 = 10 < 11`,
so the split never wins. For `n >= 12`, evaluate the recursion —
sometimes the split wins, sometimes not. This is why the memoized
function can short-circuit with `if n < 12: return n` and avoid both
recursion and cache pollution for the trivial base.
