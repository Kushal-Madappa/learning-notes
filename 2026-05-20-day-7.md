## Key concepts

### 1. Prefix sum as a *data structure* (LC #303)
Up to today, prefix sums were a one-shot trick inside a function. #303
reframes them as a **class invariant**: the constructor pays O(n) once,
every subsequent `sumRange` is O(1).

```python
class NumArray:
    def __init__(self, nums):
        self.prefix = [0] * (len(nums) + 1)
        for i, x in enumerate(nums):
            self.prefix[i + 1] = self.prefix[i] + x

    def sumRange(self, left, right):
        return self.prefix[right + 1] - self.prefix[left]
```

This is the same pattern as **Sparse Tables**, **Segment Trees**, and
**Fenwick (BIT) trees** — precompute aggressively when you know queries
will outnumber updates. The deeper insight: **amortise**. The right
question is rarely "what's the per-call complexity?" — it's "what's the
total complexity over the realistic mix of operations?"

### 2. Prefix-sum-mod-k for divisibility (LC #974)
Take yesterday's #560 and rotate the key by one degree:

```
sum(nums[l..r]) ≡ 0 (mod k)
   ⇔  P[r+1] - P[l] ≡ 0 (mod k)
   ⇔  P[r+1] ≡ P[l]  (mod k)
```

So instead of hunting for `prefix - k` in the map, you hunt for
`prefix` itself — under modular equivalence:

```python
counts = defaultdict(int)
counts[0] = 1
prefix = 0
answer = 0
for x in nums:
    prefix = (prefix + x) % k
    answer += counts[prefix]   # earlier prefixes with the same remainder
    counts[prefix] += 1
```

Subtle point: **Python's `%` always returns a non-negative result for
positive `k`**, so this works for negative inputs out of the box. In
C++/Java you'd write `((prefix % k) + k) % k`. Worth filing under
"language-specific quirks that matter in CP."

### 3. The bigger pattern — *equivalence classes of prefixes*
Once you see #974 the abstraction snaps into place: the map's key is
some **function** of the running prefix, and you're really counting
prefixes that fall into the same equivalence class under that function.

| Function of prefix | Map key | Problem class |
|--------------------|---------|---------------|
| Identity | `prefix` | Sum == k |
| Mod k | `prefix % k` | Sum divisible by k |
| Sign | `0` if zero, else `+1`/`-1` running diff | Count "more 1s than 0s" |
| XOR | `running XOR` | Subarray XOR == k |
| Even/odd of running count | `parity` | Vowel/consonant balance |

Same engine, different keys.

### 4. Working with stdin as bytes (CodeChef LUCKFOUR)
`sys.stdin.buffer.read().split()` returns a list of `bytes`, not `str`.
Two consequences:

- `data[i].count(b'4')` is a single C-level memchr scan — faster than
  the Python-level `for digit in str(n): ...` loop.
- For arithmetic you still do `int(data[i])`. Conversion cost is tiny
  next to the I/O it replaces.

This is a small thing, but at CP scale (10^5 inputs, multiple test
files), the constant matters.

### 5. Floating-point in CP — when to trust it (FLOW011)
The gross-salary formula multiplies an integer by 0.10 / 0.90 / 0.98.
With Python's IEEE-754 doubles, those products are exact enough that
`f"{gross:.2f}"` round-trips to the reference output.

For inputs and ratios this small, floats are fine. But the general
rule: **if the answer must be exact, scale into integers** (e.g.
multiply everything by 100 and stay in `int`). The day you regret
floats is the day you have an off-by-one-ULP at the boundary of a test
case. (Not today — but file the rule.)
