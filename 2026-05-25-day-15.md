## Today's work

- **LeetCode (1):**
  - **Invert Binary Tree (#226, Easy)** — the McIlroy-Whiteboard
    problem. Three lines of body: base-case on None, swap the two
    child pointers at the current node, then recurse into both
    children. Return the (now-mutated) node.
    - **The new template — mutation, not value:**
      ```python
      def f(node):
          if node is None: return None
          # 1) MUTATE at this node
          node.left, node.right = node.right, node.left
          # 2) DELEGATE to children
          f(node.left); f(node.right)
          return node
      ```
      Day 13's #104 (depth) returned a *computed value* at every
      level. Today's #226 returns the *same node* it was given,
      but the side effect along the way is what does the work.
      That's the return-value-vs-side-effect split that #543
      (Diameter) and #114 (Flatten) both lean on; getting it now
      makes those problems much smaller.
    - **Why swap-then-recurse and recurse-then-swap both work:**
      the swap touches only the two child *pointers* on the
      current node — it never reads or writes anything inside
      either subtree. So whichever order you pick, each node's
      contract is "swap my own children" and the deeper
      subtrees aren't disturbed by ordering.
    - **Inversion is its own inverse.** Invert twice and you get
      the original tree back. Useful sanity check, and the test
      harness uses it on three trees.
    - **Complexity:** O(n) time (one O(1) swap per node), O(h)
      recursion stack.
    - Verified: 3 LC samples + 4 hand-built edges + 3
      double-invert checks (10 total). All pass.

- **CodeChef (1):**
  - **Ciel and Receipt (CIELRCPT, Beginner / 1-star)** — given a
    target price `p`, the menu costs `1, 2, 4, ..., 2048` (the
    twelve powers of two from `2^0` to `2^11`), and you may order
    each menu any number of times. Find the minimum number of
    items whose prices sum to exactly `p`.
    - **Insight: greedy works *because* the denominations are
      powers of two.** Two copies of denomination `2^k` cost the
      same as one copy of `2^(k+1)`, so taking the largest
      available coin first is never worse than taking smaller
      ones. (This is the same property that makes binary
      representation unique — and the answer is exactly
      `popcount(p)` once `p` fits inside the 12-bit menu, with
      the high bits absorbed by repeated 2048s.)
    - **Algorithm (4 lines of body):** walk the denominations
      `[2048, 1024, ..., 1]` once, add `p // v` to the count,
      replace `p` with `p % v`, move on. O(12) per test —
      effectively O(1).
    - **Where this generalizes:** any "minimum number of coins
      to make change" problem where each denomination is a
      multiple of the next is greedy-solvable. As soon as that
      property breaks (e.g., US-coins-minus-the-nickel: 1, 10,
      25 — try to make 30) greedy can fail and you need DP.
      The lesson: **before reaching for DP, check whether the
      denominations are nested-multiples.**
    - Verified: official 4-test sample (`10 -> 2`, `256 -> 1`,
      `255 -> 8`, `4096 -> 2`) + 4 edge cases (single smallest,
      max constraint `p=100000 -> 52`, `p=2047 -> 11`, mixed
      batch). All pass.

## Key concepts

### 1. Two flavors of tree recursion — value vs. mutation
Day 13 installed the "return a computed value, parent combines"
template. Today installs the "mutate the current node, return the
same node so callers can chain" template. They look almost
identical on the page, but the mental model is different:

```python
# Value-returning (e.g. #104 depth, #543 diameter helper):
def f(node):
    if node is None: return BASE
    L = f(node.left)
    R = f(node.right)
    return COMBINE(L, R, node.val)

# Mutating (e.g. #226 invert, #114 flatten, #1325 delete leaves):
def f(node):
    if node is None: return None
    MUTATE(node)              # the work happens here
    f(node.left)
    f(node.right)
    return node               # almost always returns the same node
```

Hybrid problems like #543 (Diameter) actually do *both*: the
recursive helper returns the local height (value), and on the
way up it mutates a global `self.ans` with the candidate
diameter through this node (side effect). Knowing the two
templates separately makes the hybrid one obvious.

### 2. The "self-inverse" sanity check
A handful of tree/array operations are *involutions* — applying
them twice gives back the original. Inverting a tree, reversing
a list, transposing a matrix twice, complementing a bitmask.
Whenever an operation is self-inverse, a free extra test is
"call it twice, compare to input." This catches off-by-one
errors and forgotten edge cases (e.g. forgetting the empty-tree
base) for the price of zero extra reasoning. Reuse the pattern.

### 3. Coin change: greedy works iff denominations are nested-multiples
The CIELRCPT one-liner only works because every menu price is
exactly twice the next. Restate that as a rule:

> If the coin set is `c_1 < c_2 < ... < c_k` and each `c_{i+1}`
> is a multiple of `c_i`, greedy-from-largest is optimal.

Powers of two satisfy this (`2^{i+1} = 2 * 2^i`). So do US bills
(1, 5, 10, 20, 50, 100 — modulo the 5/20 awkwardness which
actually still works because of how the divisions fall out). But
the classic counterexample is `{1, 3, 4}` to make 6: greedy
picks `4 + 1 + 1 = 3` coins; optimal is `3 + 3 = 2`. **Anytime
the next-larger denomination isn't a clean multiple, drop greedy
and reach for DP.**

### 4. "Each digit in base B costs 1 coin" framing
CIELRCPT has a really clean reframing once you spot it: in base
2, the answer is just the number of 1-bits in `p`, *with* the
caveat that any "digit" above 2^11 has to be represented as
multiple copies of the 2048 coin (so add `p // 2048` and then
popcount the remainder). The same trick works any time the coin
set is `{B^0, B^1, ..., B^{k-1}}` for some base B and a cap k —
the answer becomes "digit sum of p in base B, with overflow
absorbed by k-1 copies of B^{k-1}". File that under "problems
that look like search but are actually arithmetic."
