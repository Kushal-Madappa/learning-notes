## Key concepts

### 1. Prefix sum as a baseline (LC #724)
The prefix-sum mental model: imagine a parallel array
`P[0] = 0`, `P[i] = nums[0] + ... + nums[i-1]`. Then
`sum(nums[l..r]) = P[r+1] - P[l]` — any range sum becomes O(1).

For #724 we don't even need the array — only the running
left sum and the total:

```python
total = sum(nums)
left = 0
for i, x in enumerate(nums):
    if left == total - left - x:
        return i
    left += x
```

That `total - left - x` is the right sum derived on the fly. The whole
problem collapses to a single pass with one extra integer.

### 2. Prefix sum + hash map = "count subarrays with sum k" (LC #560)
This is one of the highest-leverage tricks in interview problem solving.

```python
counts = defaultdict(int)
counts[0] = 1          # empty-prefix sentinel
prefix = 0
answer = 0
for x in nums:
    prefix += x
    answer += counts[prefix - k]   # how many earlier prefixes leave us at k?
    counts[prefix] += 1
```

The key reframing:

    sum(nums[l..r]) == k
       ⇔  P[r+1] - P[l] == k
       ⇔  P[l] == P[r+1] - k

So at every right endpoint, the number of valid left endpoints is
exactly the number of *earlier* prefix sums equal to `P[r+1] - k`.
A hash map counts those in O(1).

The `counts[0] = 1` seed is what lets subarrays that *start at index 0*
get counted — those correspond to `P[l] = 0`, the empty prefix.

**Why sliding window doesn't work here:** sliding window requires the
"shrinking the window monotonically improves the metric" invariant.
With negative numbers, shrinking can *increase* the sum, breaking
monotonicity. Prefix sums sidestep the issue entirely.

### 3. Closed-form vs. loop (TRICOIN)
Both work, but they teach different muscles:

```python
# Closed form — O(1), the CP idiom
H = (math.isqrt(8 * n + 1) - 1) // 2

# Loop — O(√n), more transparent
h = 0
while (h + 1) * (h + 2) // 2 <= n:
    h += 1
```

The closed form uses `math.isqrt` (added in Python 3.8) which is the
**integer** square root — no float, no rounding bugs. Using `math.sqrt`
here would silently fail for very large N because of float precision.

Mental note: any time a problem says "find the largest k such that
some polynomial in k is ≤ N", reach for either binary search or a
closed form.

### 4. Python's hidden CP superpower — big integers (FCTRL2)
In C++ or Java, 100! requires implementing a `BigInt` (or using
external libraries). In Python, `math.factorial(100)` just works and
prints all 158 digits.

Languages aside, the lesson is: **know which native types of your
language are unboundedly large**. In Python: `int`. In Java: `BigInteger`.
In C++: implement it or use `__int128` for narrow gains.

### 5. The "running totals + map" family — what to spot next
| Pattern | Map key | Use cases |
|---------|---------|-----------|
| Prefix sum + count map | running sum | count subarrays with sum k (#560) |
| Prefix sum + first-index map | running sum | longest subarray with sum k (#325) |
| Prefix sum mod m + map | running sum mod m | count subarrays divisible by m (#974) |
| Prefix XOR + map | running XOR | count subarrays with XOR k (#1442) |

Once you see one, you'll see them everywhere — they all use the same
"register the running value, query for the complement" skeleton.

## Patterns to spot next
- **Subarrays Divisible by K (#974, Medium)** — same prefix-sum +
  hash-map skeleton, but the key is `prefix % k`. Negative-mod
  normalisation matters: `(prefix % k + k) % k`.
- **Continuous Subarray Sum (#523, Medium)** — variant that stores
  the *first index* of each remainder, not the count.
- **Binary Search bridge** — once range-sum queries are O(1), many
  problems pair prefix sums with binary search (e.g. "smallest subarray
  with sum ≥ s"). This is the next major topic.
