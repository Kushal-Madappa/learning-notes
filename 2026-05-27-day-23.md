### LeetCode (1)

1. **Combination Sum IV (#377, Medium)** — given a list of distinct
   positive integers `nums` and an integer `target`, return the
   number of **ordered sequences** of elements of `nums` (with
   repetition allowed) summing to `target`.
   - **The name lies.** Despite "Combination" in the title, the
     problem counts permutations — `[1, 2]` and `[2, 1]` are two
     answers, not one. The LeetCode example
     `nums=[1,2,3], target=4 -> 7` makes that explicit (`1+1+1+1`,
     `1+1+2`, `1+2+1`, `2+1+1`, `2+2`, `1+3`, `3+1`).
   - **Recurrence:** `dp[a] += dp[a - c]` for every coin `c <= a`,
     with `dp[0] = 1`. Identical to #518's recurrence body.
   - **What changes — the loop nesting.** Outer = amount, inner =
     coins.
     ```python
     for a in range(1, target + 1):
         for c in nums:
             if c <= a:
                 dp[a] += dp[a - c]
     ```
     The right-hand `dp[a - c]` already represents *every* way to
     make `a - c` regardless of which coin was used last. So
     appending coin `c` produces a new sequence whose *last element*
     is `c` — and the same multiset is reached once per last-element
     choice, giving the permutation count.
   - **Why swapping the loops flips the meaning.** In #518's
     outer-coins / inner-amount order, by the time we update `dp[a]`
     with coin `c`, the right-hand `dp[a - c]` only reflects
     coins `<= c` in the iteration order — so each multiset is
     reached exactly once, and the count is combinations.
   - **Base case `dp[0] = 1`** is the empty-product convention again:
     one way to make zero (use nothing). Same as #518, opposite of
     #322 where `dp[0] = 0` because the aggregation is "min coins".
   - **Follow-up trap LeetCode flags:** what if negatives are allowed
     and a coin can be subtracted? Then any reachable amount becomes
     reachable infinitely many ways, and the count is undefined. The
     standard fix is to bound the sequence length and add a second
     dimension `dp[a][k]` over length. (Not implemented here — but
     worth being able to articulate.)
   - **Complexity:** Time O(target × len(nums)), Space O(target).
   - Verified across 10 cases including the LC samples
     (`[1,2,3], 4 -> 7`; `[9], 3 -> 0`), the empty-target edge
     (`[1,2,3], 0 -> 1`), parity-unreachable (`[2], 5 -> 0`),
     length stress (`[1,2,3], 10 -> 274`), and the
     direct contrast `nums=[1,2,5], target=5 -> 9` (vs. 4 for
     #518 with the same inputs).

### CodeChef (1)

1. **IPL (ZCO14004, Easy-Medium)** — `N` consecutive matches, each
   with a viewing fee/profit `a[i]`. Pick a subset to **maximise
   total profit**, subject to: cannot pick 3 or more consecutive
   matches. (Mirror of SUPW, which *minimised cost* subject to
   cannot *skip* 3 or more in a row.)
   - **Recurrence:** let `dp[i]` = best profit using prefix `[0..i]`.
     ```
     dp[i] = max(
         dp[i-1],                       # don't pick match i
         dp[i-2] + a[i],                # pick i alone (skip i-1)
         dp[i-3] + a[i] + a[i-1]        # pick i and i-1 (skip i-2)
     )
     ```
     The third term is the line that distinguishes IPL from the
     naive #198-style "pick or skip" DP. Without it, you cannot
     accommodate the legal *PP_* pattern (two consecutive picks
     followed by a forced skip).
   - **Bases:** `dp[-1] = 0` (treat as out of range), `dp[0] = a[0]`,
     `dp[1] = a[0] + a[1]` (both legal — only 3 in a row forbidden,
     so 2 in a row is fine and trivially optimal for `n=2`).
   - **The forum's canonical breaking test:** `n=10`,
     `a = [3, 2, 1, 1, 2, 3, 1, 3, 2, 1]` → `15`. A naive
     two-transition DP (without the `dp[i-3] + a[i] + a[i-1]` term)
     returns `14`. This is the classic "consider all 3 predecessors,
     not just 2" lesson.
   - **Final answer is `dp[N-1]`** (not `min/max` over a suffix the
     way SUPW was). The reason: `dp[i]` here is **best over prefix**,
     not "best assuming `i` is picked." That subtle reframing
     between SUPW and IPL is worth holding in mind — they're the
     same family but the *definition* of `dp[i]` is different.
   - **Why O(1) rolling space still works:** the recurrence still
     only ever looks back 3 positions, regardless of how `dp[i]` is
     defined. Three rolling scalars (`d_m3, d_m2, d_m1`) suffice.
   - **Complexity:** Time O(N), Space O(1).

## Key concepts

### 1. The loop-order matrix — one recurrence, four behaviours

Today's #377 vs. yesterday's #518 nails down a 2×2 picture of
unbounded-coin DPs that's worth keeping on a flashcard:

| Aggregation \ Loop order | outer coins / inner amount | outer amount / inner coins |
|--------------------------|----------------------------|----------------------------|
| **`+=` (count)** | **combinations** (#518)    | **permutations** (#377)    |
| **`min` / `max`** | order-insensitive (works either way for #322 min-coins) | order-insensitive (works either way for #322 min-coins) |

The bottom row is a freebie: for `min`/`max` aggregations on a
1-D `dp[amount]` table, the loop order doesn't matter because both
nestings consider every `(coin, amount)` pair, and the aggregator
is associative/commutative. The interesting row is the top one:
*`+= ` cares about order because addition counts incidents, not
sets.* That's the whole game.

The reframe that finally made #518 vs. #377 click: when you say
`dp[a] += dp[a - c]`, ask "in what *order* did I commit to coins
along the way to `dp[a - c]`?" If the iteration only allowed coins
processed *up to now* in the outer loop, you committed to a sorted
order — that's a multiset / combination. If `dp[a - c]` saw every
coin freely, you committed to whatever coin was *last* — that's a
sequence / permutation.

### 2. "Best-over-prefix" vs. "best-ending-at-i" — the same family

SUPW's `dp[i]` was "best cost *with `i` as the last picked day*"
— so the final answer was `min(dp[N-3], dp[N-2], dp[N-1])` to
guard the trailing window. IPL's `dp[i]` is "best profit *over
prefix `[0..i]`*" — so the final answer is just `dp[N-1]`.

Both definitions work; both yield O(N) DPs; the recurrences look
nearly identical. **The difference is at the boundary.** A useful
mental check whenever you're staring at a "look back k steps" DP:

> Did I define `dp[i]` as *conditional on `i` being picked*, or as
> *best so far over the prefix*? If conditional, the final answer
> is an aggregation over the last `k` indices. If prefix, the
> final answer is `dp[N-1]`.

Pick whichever framing makes the recurrence cleaner — sometimes
the conditional framing avoids ugly "or skip i" cases (LIS #300),
sometimes the prefix framing avoids the trailing-suffix aggregation
(IPL). Both are fine; just stay consistent within one problem.

### 3. The "third transition" trap

The IPL discussion forum is full of beginners with code that
considers only `dp[i] = max(dp[i-1], dp[i-2] + a[i])`. That two-
transition DP is the #198 House Robber template (no two
consecutive picks), and it fails IPL because IPL **allows two
consecutive picks** — the constraint is no *three* in a row. The
missing transition is `dp[i-3] + a[i] + a[i-1]` (the PP at the
right edge, the skip three steps back).

The lesson generalises: whenever the constraint is "no `k` in a
row," the recurrence has `k - 1` transitions (skip i, take i,
take i and i-1, ..., take i through i-k+2). #198 is `k=2` so two
transitions. IPL is `k=3` so three transitions. The pattern
extends linearly.
