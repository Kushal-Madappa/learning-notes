## Today's work

- **LeetCode (1):**
  - **Capacity to Ship Packages Within D Days (#1011, Medium)** — the
    binary-search twin of Koko Eating Bananas (#875). Same five-line
    parametric template; the only thing that changes is the predicate
    and the bounds.
    - **Answer space:** capacity `c ∈ [max(weights), sum(weights)]`.
      - `max(weights)` is the hard floor — anything less can't fit the
        heaviest single package.
      - `sum(weights)` is the trivial ceiling — load everything in one
        day.
    - **Predicate `can_ship(c)`:** one-pass greedy. Maintain `(used,
      load)`. For each weight `w`, if `load + w > c`, finish the
      current day (`used += 1`) and start a fresh day with `load = w`;
      else accumulate `load += w`. Return `used <= days`.
    - **Monotonicity:** more capacity never needs more days, so the
      `False/True` boundary exists and is unique → `lower_bound` finds
      the minimum feasible capacity.
    - Verified: 3 official LC samples + 4 hand-built edges (1 day, N
      days, single package, equal weights) + brute-force cross-check
      on 200 random small inputs.

## Key concepts

### 1. The parametric-binary-search arc is closed
Across Days 8–12 the same template solved Sqrt(x) (#69), Valid Perfect
Square (#367), First Bad Version (#278), Search Insert Position (#35),
Koko Eating Bananas (#875), and now Ship Packages (#1011). The
template is now reflex:

```python
def feasible(x):  # monotone predicate over the answer space
    ...

lo, hi = <lower bound>, <upper bound>
while lo < hi:
    mid = (lo + hi) // 2
    if feasible(mid):
        hi = mid
    else:
        lo = mid + 1
return lo
```

The only mental work per problem is now:

1. **Identify the answer space** — what range does the answer live in?
2. **Write the predicate** — one function `feasible(x): bool` that
   answers "is `x` good enough?"
3. **Confirm monotonicity** — once `feasible` is True, it stays True
   for larger `x` (or smaller, depending on direction).

Everything else is the same loop. That's the entire point of
"parametric": the search engine doesn't change, only the predicate.

### 2. Ship-packages vs. Koko: structural identity, surface difference
Koko's predicate was `hours(k) = Σ ⌈p / k⌉ ≤ h` — closed-form,
arithmetic. Ship's predicate is `can_ship(c)` — a stateful one-pass
walk that simulates loading days. Both are O(N), both are monotone in
the parameter. The fact that one looks like "math" and the other like
"a tiny algorithm" is cosmetic; the binary search doesn't care.

When predicates look this different but the engine is identical,
that's the strongest signal that you've internalised the pattern:
you stopped pattern-matching on syntax and started pattern-matching
on shape.

### 3. The greedy in `can_ship` is optimal — and it's worth knowing why
The "start a new day whenever adding the next package overflows" rule
is greedy, and greedy proofs are usually where students get sloppy.
Here it's clean: packages must be shipped **in order**, so the only
decision per day is *where to break*. Breaking later is never worse
than breaking earlier — if you can fit one more package today, doing
so leaves an equal or smaller residual for tomorrow. So the latest-
break strategy minimises days for any fixed capacity. That's the
"exchange argument" in one sentence.

This matters because it tells you `can_ship` is *exact*, not an
approximation — so the answer to the binary search is the true
optimum, not a heuristic upper bound.

### 4. Fast I/O budget on CodeChef Python: the actual numbers
TSORT is the canonical case for `sys.stdin.buffer.read().split()`.
With N = 10⁶ integer reads, the per-line cost of `input()` dominates
the entire runtime. Rough order-of-magnitude on CPython:

| Read method                          | ~ns per int  |
|--------------------------------------|--------------|
| `int(input())` in a loop             | ~700–1000    |
| `int(sys.stdin.readline())` in loop  | ~300–400     |
| `sys.stdin.buffer.read().split()`    | ~80–120      |

At 10⁶ ints, the difference is the difference between "AC in 1.2 s"
and "TLE at 8 s." This is exactly what the Day-11 heuristic table
predicted — TSORT is in the third row, no judgment call needed.

## Patterns to spot next (Trees week)

The trees arc starts tomorrow. The mental shift is from "1-D search"
to "branching state." A few problems to look for:

- **Maximum Depth of Binary Tree (#104, Easy)** — the "hello world"
  of recursion on trees. `depth(node) = 1 + max(depth(left),
  depth(right))`, base case `depth(None) = 0`. Builds the recursive
  reflex.
- **Same Tree (#100, Easy)** — structural recursion on two trees
  in parallel. Pattern: `same(a, b) = (a is None and b is None) or
  (a and b and a.val == b.val and same(a.left, b.left) and
  same(a.right, b.right))`.
- **Invert Binary Tree (#226, Easy)** — the legendary interview
  question. One line of recursion + a swap. The point is that the
  *return* of a recursive call is the inverted subtree, so you can
  reassign `node.left, node.right = invert(node.right),
  invert(node.left)` and you're done.
- **Binary Tree Inorder Traversal (#94, Easy/Medium)** — the
  recursive version is 4 lines; the iterative-with-stack version is
  the one to actually learn, because it generalises to BST
  validation, k-th smallest, and Morris traversal later.
- **Level Order Traversal (#102, Medium)** — first BFS problem.
  Queue with a "size of current level" trick to emit one list per
  level. The same engine will reappear for #199 (right side view)
  and #515 (largest value per level).
