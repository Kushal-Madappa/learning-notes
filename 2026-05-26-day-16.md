- **CodeChef (1):**
  - **Ciel and A-B Problem (CIELAB, Beginner / 1-star)** — given
    `A` and `B`, print *any* positive integer with the same digit
    count as `A - B` and exactly one digit different. Constructive
    problem: there are many valid outputs; pick the cheapest.
    - **Insight: just tweak the last digit.** The whole problem
      collapses to a single ternary:
      ```python
      print(c - 1 if c % 10 == 9 else c + 1)
      ```
      Adding 1 to the last digit normally changes exactly one
      digit and preserves the digit count — the one case where
      that breaks is when the last digit is `9` (because `+1`
      carries and changes at least two digits, and at the top of
      the range adds an extra digit). In that single case, do
      `c - 1` instead.
    - **Why "be careful not to print 0 if A-B = 1" doesn't bite
      this solution:** when `c = 1`, `c % 10 = 1`, so we hit the
      `c + 1 = 2` branch — never the `c - 1 = 0` branch.
    - **Algorithm:** 3 lines of body. O(1) per test case.
    - Verified: official sample (`5858 1234` -> expects any
      4-digit one-off from 4624; our `4625` qualifies) + 8 edge
      cases (`1-0`, `10-1`, `100-1`, `1000-100`, `1e9-1`, etc.).
      The verifier checks: same digit count, exactly one differing
      digit, positive, no leading zero. All pass.

## Key concepts

### 1. The two-tree template, both knobs visible
After three days of two-tree problems (#100, #226, #101), the
template is now fully exposed. Every two-tree problem you'll see
in the easy/medium range fits this shape:

```python
def f(a, b):
    if a is None and b is None: return BASE_BOTH_NONE
    if a is None or  b is None: return BASE_ONE_NONE
    if not LOCAL_OK(a, b):      return False
    return f(a.LEFT_LIKE, b.RIGHT_LIKE) and f(a.RIGHT_LIKE, b.LEFT_LIKE)
```

Two knobs:
- **`LOCAL_OK`** decides what "agreement at this node" means.
  Value-equality (#100, #101), subtree-equality (#572), filtered
  child match (#1325).
- **The pairing** decides what "next step" means. Parallel
  `(left, left)+(right, right)` for #100 ("same"); crossed
  `(left, right)+(right, left)` for #101 ("mirror"). Some
  problems use a *mixed* pairing (e.g., #872 leaf-similar trees
  collapses to a single sequence, sidestepping the question).

Once you can name the two knobs, most two-tree easy/mediums turn
into a 5-minute exercise: "what's LOCAL_OK, what's the pairing,
done." File this for #572, #1325, and the eventual #951 Flip
Equivalent Binary Trees, which is literally "either pairing works
at each level — return True if either pairing succeeds."

### 2. Constructive problems vs. computational problems
CIELAB is the first **constructive** problem in this profile —
the task is "find *any* valid output," not "compute *the* answer."
That changes the strategy completely:

- **Computational problems** ("what is the A-B?"): you must
  reproduce a specific value. Any deviation is wrong.
- **Constructive problems** ("give me *any* wrong A-B"): you
  pick the cheapest valid construction. Don't search the answer
  space, just exhibit one element.

The mental move for constructive problems is: **make the smallest
change possible and verify it satisfies the constraints**. For
CIELAB, "smallest change" = one digit; "constraints" = same digit
count + exactly one digit different + positive + no leading zero.
The `+1`/`-1` trick is the smallest change that satisfies all of
them.

This pattern recurs in many CodeChef beginner problems and a
handful of LeetCode constructive problems (#1947 Maximum
Compatibility Score Sum, #1968 Array With Elements Not Equal to
Average of Neighbors). The reflex is: "is this 'find the answer'
or 'find any answer?' If it's any-answer, the cheapest
construction is almost always the right move."

### 3. The "boundary 9" trap (and how to spot it)
The `c + 1` trick for CIELAB breaks at exactly one place: the
last digit being `9`. That's a *boundary-of-digit-range* bug, and
the general lesson is broader than this problem:

> Any time your algorithm changes a digit by +/- 1, you need to
> handle the boundary case where that change carries.

This same flavor of bug shows up in:
- **Binary representations**: flipping a bit might cascade
  (1011 + 1 = 1100, three bits flipped).
- **Date arithmetic**: adding 1 day at month-end / year-end.
- **Counter overflow**: any fixed-width counter at its max.

The mitigation pattern is what we used here: **split into the
non-boundary case (most of the input) and the boundary case (one
specific digit / state)**, and handle each with a one-line rule.
If your code has a `+1` somewhere and you didn't think about the
boundary, that's the bug.

### 4. Multi-line vs. one-line CIELAB-style solutions
The minimal version is one expression — `print(c - 1 if c % 10
== 9 else c + 1)`. That's three operations: subtract, mod, branch.
But it's worth knowing the *general* construction in case the
constraints change:

> For "change one digit, keep digit count," you can always pick
> the *first* digit if it's not 1 (subtract 1 from it) or the
> first digit if it's 1 (add 1 to it). The first-digit trick
> never carries and never adds a digit, so it's the most robust
> construction. The last-digit trick is shorter but has the
> boundary case.

When you see a constructive problem with multiple valid moves,
keep the most robust construction in your back pocket even if you
ship a slicker one.
