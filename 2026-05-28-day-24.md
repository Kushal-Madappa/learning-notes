## Today's work

### LeetCode (1)

1. **Number of Islands (#200, Medium)** — given an `m x n` grid of
   `'1'` (land) and `'0'` (water), return the number of islands. An
   island is a 4-directionally connected set of `'1'`s.
   - **The graph in disguise.** No edges are given; adjacency is
     defined by the four cardinal neighbours of each cell. The
     "graph" has `m*n` nodes and up to `4*m*n` directed edges, but
     we never materialise it — we walk the grid directly.
   - **Algorithm — flood fill on first hit.** Scan every cell. When
     we see a `'1'` we've never visited, that *must* be the start of
     a previously unseen island (because any cell in a known island
     would already be marked visited), so we increment the counter
     and run a flood fill that paints the whole component visited.
   - **Why marking in place works.** Rewriting visited `'1'`s to
     `'0'` is the cheapest possible "visited set" — it costs zero
     extra memory and lets the outer loop's `grid[r][c] != "1"`
     guard double as "already counted." If the caller cares about
     the original grid, copy first.
   - **Why iterative DFS, not recursive.** A worst-case snake-shaped
     island can be up to `m*n` cells long. On LeetCode's grids
     (up to 300×300 = 90 000 cells) Python's default recursion
     limit (1000) would blow up. An explicit stack (`deque` for the
     `pop()` speed) sidesteps the issue entirely.
   - **BFS vs DFS — both work, same complexity.** The choice only
     matters when you need *shortest* path (BFS) or when memory
     profile matters (BFS uses a wider queue, DFS a deeper stack).
     For *counting components*, either is fine.
   - **Edge cases worth holding in mind:** empty grid `[]`, empty
     row `[[]]`, all water (`0` islands), all land (`1` island), a
     donut-shaped island with a separate island in the hole (`2`
     islands — the outer ring and the interior cell are not
     connected through the water moat). All exercised in the local
     test block.
   - **Complexity:** Time `O(m * n)` — each cell is pushed and
     popped at most once. Space `O(m * n)` worst case for the stack
     on a pathological snake-shaped island.
   - **Where this template appears next:** #695 (Max Area of
     Island — count cells instead of components), #463 (Island
     Perimeter — count "outside" edges per land cell), #1254
     (Closed Islands — filter out islands touching the border via
     a pre-pass that floods from the edges), #130 (Surrounded
     Regions — same border pre-flood idea, applied differently),
     #547 (Number of Provinces — same component-counting pattern
     but on a true adjacency matrix). The grid-flood-fill template
     written today is reused, near-verbatim, in all of them.

### CodeChef (1)

1. **AMSGAME1 — Chef and Chocolates / Subtraction Game 1
   (Beginner / Cakewalk)** — `T` test cases, each with `N`
   positive integers. The game: at each step pick two **unequal**
   numbers and replace the larger by the difference. Stop when all
   numbers are equal. Report the stopping value.
   - **The whole problem collapses to one identity.** The
     operation `(a, b) -> (a-b, b)` (with `a > b`) preserves
     `gcd(a, b)`, because `gcd(a, b) = gcd(a - b, b)` — this is
     the recursion step of the **Euclidean algorithm**. Inductively,
     the gcd of the entire multiset is invariant under every legal
     move. Since the process must terminate at `(k, k, ..., k)`
     where `gcd(k, k, ..., k) = k`, the stopping value `k` equals
     the original gcd `G`.
   - **Therefore the answer is `reduce(gcd, A)`** — no need to
     simulate, no need to think about which two numbers to pick or
     in what order.
   - **Python implementation in 11 lines.** `from math import gcd`,
     fold with an early break when `g == 1` (since gcd with 1 is
     always 1 and the answer can't shrink further).
   - **Why the early-exit `if g == 1: break` matters.** With
     `N` up to ~10^3 and values up to ~10^9, the work is trivially
     fast either way, but in adversarial inputs (e.g., `[..., 1,
     ...]`) the loop is `O(1)` instead of `O(N)`. Good habit to
     carry into harder GCD problems where N can be 10^5+.
   - **Complexity:** Time `O(N log V)` per test case where V is the
     max value (each gcd call is logarithmic via Euclidean).
     Space `O(1)` beyond input.
   - **The shape of this problem.** The wording disguises a
     process / game / simulation, but the answer is *a property of
     the input that is invariant under the process*. Whenever you
     see "perform operations until ___, report the final value,"
     ask: **what quantity is unchanged by every operation?** That
     quantity is your answer. Same trick:
     - **Bitwise XOR of all elements** is invariant under
       "delete two equal numbers" (Single Number, LC #136).
     - **Sum** is invariant under "swap two elements"
       (almost all rearrangement problems).
     - **Parity / count modulo k** is invariant under many
       move-pair-then-replace operations.

## Key concepts

### 1. The grid-as-graph mental model

A 2D grid with 4-directional adjacency is just a graph where:
- nodes are `(row, col)` pairs,
- edges connect each cell to its in-bounds N/S/E/W neighbours,
- "connected component" = "island."

Once that mapping clicks, every graph algorithm carries over:
**count components** (today), **shortest path on the grid**
(BFS, LC #1091 / #994), **detect cycle in a maze** (DFS with
recursion stack), **bipartite check** (alternating BFS colours),
**reachability with edge weights** (Dijkstra on grid). The grid
just gives you the adjacency for free — the algorithms are
identical to the ones you'd run on an adjacency list.

### 2. Iterative DFS vs recursive DFS — when to switch

Recursive DFS is shorter to write but fails on deep components:

| Concern | Recursive DFS | Iterative DFS (stack) |
|---------|---------------|------------------------|
| Code length | ~5 lines | ~10 lines |
| Recursion limit | Crashes at ~1000 deep on default | No limit (heap) |
| Speed (Python) | Slower (function call overhead) | Faster |
| Debuggability | Stack trace shows path | Need to log explicitly |

Rule of thumb: for grids with `m * n > 1000` cells, or for graphs
where a long chain is plausible, **default to iterative**. The
extra few lines are cheap insurance.

### 3. Marking visited *in the input* — when it's OK

Rewriting `grid[r][c] = "0"` to mark a visited cell is the
zero-memory visited set. It's safe when:
1. The caller doesn't need the original grid back, **or**
2. You document that the function mutates input.

If neither holds, use a separate `set` of `(r, c)` pairs or a
`visited` 2D array. Common bug in LeetCode submissions: solving
two problems on the same grid and finding the second one returns
`0` because the first solved-in-place pass wiped everything to
water.

### 4. Invariants as solutions — the AMSGAME1 pattern

When a problem describes a *process* ("repeat this operation
until ___, report the final value"), the answer is often **a
quantity unchanged by the operation**. The technique:

1. Identify the operation precisely (`(a, b) -> (a-b, b)` here).
2. Find a property `P` such that `P(state_before) = P(state_after)`
   for every legal move.
3. Evaluate `P` on the *final* state (which is usually
   constrained — here, "all equal" forces `P = k`).
4. Therefore `P(initial input) = k` — that's the answer.

For AMSGAME1 the invariant is `gcd`. For XOR-pair problems it's
the XOR-fold. For sum-preserving rearrangements it's the sum.
For coin-flipping problems it's often `sum mod 2`. **Worth
keeping this lens active** — it converts simulation problems into
one-line answers.

### 5. Euclidean algorithm and `math.gcd`

`math.gcd(a, b)` in Python uses an optimised binary GCD (Stein's
algorithm) under the hood; for our purposes it's `O(log min(a,b))`
per call. To extend to a list: `from functools import reduce;
reduce(gcd, nums)`. For three or more numbers,
`gcd(a, b, c) = gcd(gcd(a, b), c)` — that associativity is what
lets the `reduce` work.

Useful identities to keep in pocket for future problems:
- `lcm(a, b) = a * b // gcd(a, b)` — be careful with int
  overflow in non-Python languages.
- `gcd(0, n) = n` — handy when an array contains zeros.
- `gcd(a, b, c, ...) = 1` iff the integers are **coprime as a
  set** (not necessarily pairwise coprime).
