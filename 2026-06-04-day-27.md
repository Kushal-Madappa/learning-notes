## Today's work

### LeetCode (1)

1. **Rotting Oranges (#994, Medium)** — given an `m x n` grid of
   `0` (empty), `1` (fresh orange), `2` (rotten orange). Every
   minute, any fresh orange that shares an edge with a rotten one
   becomes rotten. Return the minimum minutes for all fresh oranges
   to rot, or `-1` if any will never rot.
   - **Why this needs BFS, not DFS.** The question "how many
     minutes" is a *shortest-time* query: we need the minimum number
     of rot-propagation steps until everything is rotten. DFS would
     find *a* path; BFS gives the *shortest* path from the source
     set to every reachable node — which is exactly what we want.
     The pattern: **if the question is "minimum steps / shortest
     distance / fewest minutes," use BFS.** If it's "is X reachable
     / how many components / max area," DFS is equally fine.
   - **Multi-source BFS — the key twist.** Single-source BFS seeds
     the queue with one node and computes distance from it. Here
     we have *multiple* rotten cells at time 0, and we want the
     shortest distance from the *nearest* rotten cell to each fresh
     one. The clean way to express this: **seed every source into
     the queue simultaneously with distance 0.** BFS then naturally
     finds the shortest distance from the source *set*, treating
     all sources as if they were merged into a single super-source.
     This is exactly the physical model: every minute, every
     rotten cell rots its neighbours — the wavefronts of multiple
     fires advance in lockstep.
   - **Tracking the time / level.** Two equivalent ways:
     1. Per-node triples `(r, c, time)` — every pushed neighbour
        gets `time + 1`. Track `max_time` (or equivalently the
        last popped time, since BFS pops in time order).
     2. Level-by-level: at the top of each iteration, snapshot
        `levels_at_this_depth = len(queue)`, then pop exactly that
        many, then `minutes += 1`. The outer-loop counter is the
        depth.
     I used (1) here — slightly less code; the depth falls out of
     the triple itself. Use (2) when you also need to operate on
     "everything at this level" as a batch (e.g., level-order
     tree traversal #102).
   - **Detecting unreachable fresh oranges.** Count fresh up front;
     decrement as each one rots. If `fresh > 0` after BFS drains
     the queue, those cells were never reached → return `-1`.
     **Don't try to detect by checking "did anything change this
     round"** — that confuses the no-fresh case (return 0) with the
     all-isolated case (return -1).
   - **Edge cases that catch bugs:**
     - **No fresh oranges** → return 0 (NOT the BFS result, which
       might be 0 anyway, but the early-exit makes the intent
       explicit and avoids accidentally returning -1).
     - **No rotten oranges, fresh present** → BFS queue starts
       empty, loop doesn't run, `fresh > 0` at end → return -1.
     - **Single rotten cell in 1xN row of fresh** → exactly `N-1`
       minutes.
     - **Multiple rotten cells at opposite ends** → minutes equals
       `ceil(distance_between / 2)` — the BFS frontiers meet in the
       middle. This is the *whole point* of multi-source BFS.
     - **Diagonal-only adjacency** doesn't count → fresh orange
       diagonally adjacent to a rotten one is unreachable.
   - **Complexity:** Time `O(m * n)` — each cell is enqueued at
     most once. Space `O(m * n)` — worst case the queue holds all
     cells (everything rotten at t=0).
   - **Template ready for nearby problems:**
     - **#1091 Shortest Path in Binary Matrix** — single-source
       BFS on a grid with 8-directional adjacency. Same engine,
       different `DIRS`.
     - **#542 01 Matrix** — multi-source BFS where the sources
       are *all* `0` cells and we ask the distance to the nearest
       `0` for every `1`. Identical to today's pattern, swap
       "rotten" for "zero" and the return type changes from a
       single number to a distance grid.
     - **#1162 As Far from Land as Possible** — multi-source BFS
       seeded from all `1`s, return the max BFS depth. Today's
       template, return `elapsed` instead of `0` even when fresh
       is fully processed.

### CodeChef (1)

1. **GCD2 (Easy)** — `T` test cases; each line has two non-negative
   integers `A` and `B` where one of them can be up to **250 digits**.
   Print `gcd(A, B)` for each.
   - **The "trick" — and why Python sidesteps it.** In C/C++ the
     250-digit number doesn't fit in `long long` (which caps around
     19 digits). The canonical approach is to read the big number
     as a string and compute `big mod small` digit-by-digit
     (Horner-style: `r = (r * 10 + (digit)) % small`), then call
     Euclid on `(small, big_mod_small)`. In Python, **`int(s)`
     parses an arbitrary-length numeric string into a native
     unlimited-precision int**, and `math.gcd` accepts any
     non-negative integers, so the entire problem becomes:
     ```python
     a, b = map(int, input().split())
     print(gcd(a, b))
     ```
     This is one of the cleanest demonstrations of "Python's
     bignum is a competitive-programming superpower" — see also
     **FCTRL2** (compute `N!` for `N` up to 100 = up to ~158 digits;
     in Python: `print(math.factorial(n))`).
   - **`math.gcd(0, n)` returns `n`** — handy for the edge case
     where the small number is 0 and the answer is the big number
     itself.
   - **`math.gcd` is the C-implemented Euclidean GCD.** Even for
     250-digit operands, the first call reduces the operand to
     at most ~10 digits (since `big mod small` is bounded by
     `small`), and the remaining log-step Euclidean iterations are
     trivially fast. So the perceived hardness of GCD2 is entirely
     a function of your language's number representation, not the
     algorithm itself.
   - **Complexity:** Time `O(log min(A, B) + d)` where `d` is the
     digit count of the larger number (paying for the bignum
     read+mod step once). Space `O(d)` for the bignum itself.

## Key concepts

### 1. BFS vs DFS — the cheat sheet

After three days of DFS and one day of BFS, the decision rule has
crystallised:

| Question                                  | Use     | Why                                                                     |
|-------------------------------------------|---------|-------------------------------------------------------------------------|
| Is X reachable from Y?                    | Either  | Both visit every node in a component; BFS terminates earlier if X close |
| How many connected components?            | Either  | The skeleton is identical; pick the one that's simpler in code          |
| Shortest path / fewest steps (unweighted) | **BFS** | DFS doesn't guarantee shortest; BFS does, by visit order                |
| Longest path in a DAG                     | **DFS** | Recursion-with-memo on topological order                                |
| Cycle detection in undirected             | Either  | Track parent during DFS, or "did we touch a visited non-parent" in BFS  |
| Cycle detection in directed               | **DFS** | Three-colour states (white/grey/black) require recursion stack          |
| Topological sort                          | **DFS** | Append on post-order; reverse                                           |
| Level-by-level processing (tree/graph)    | **BFS** | Each loop iteration = one level; trivial to batch per-level work        |
| "Minimum time until everything reaches X" | **BFS** | Multi-source seeded at all targets; one big wavefront                   |
| Tree DP (subtree results aggregate up)    | **DFS** | The recursion's return value *is* the answer per subtree                |

**Today's #994 is the canonical multi-source BFS problem.** When
you see "every X simultaneously affects neighbours per minute,"
that's the signal.

### 2. Multi-source BFS — the seed-the-queue trick

The pattern in three lines:

```python
queue = deque()
for src in sources:
    queue.append((src, 0))     # all sources at distance 0
    visited[src] = True
while queue:
    node, dist = queue.popleft()
    for nbr in neighbours(node):
        if not visited[nbr]:
            visited[nbr] = True
            queue.append((nbr, dist + 1))
```

The mathematical claim: this computes, for every reachable node,
the *minimum* distance to *any* source. Equivalent to adding a
virtual super-source connected to every real source by a 0-cost
edge, then running single-source BFS from the super-source.
The "0-cost" only works because BFS uses edge count, not weight
— for weighted graphs you'd need Dijkstra with the same
virtual-super-source idea.

Use cases beyond Rotting Oranges:
- **Distance to nearest land** (#1162, #542 — invert: distance
  to nearest 0).
- **Multi-target shortest path on a grid** — seed all targets,
  let one BFS compute distance to nearest target for every
  cell.
- **Fire / disease / water spread simulations** — every initial
  ignition point is a source.

### 3. Level-by-level BFS vs node-by-node BFS

Two equivalent BFS templates:

**Triple-with-depth (what I used today):**
```python
queue = deque([(start, 0)])
while queue:
    node, depth = queue.popleft()
    # process node
    for nbr in neighbours(node):
        queue.append((nbr, depth + 1))
```

**Level-snapshot:**
```python
queue = deque([start])
depth = 0
while queue:
    for _ in range(len(queue)):   # snapshot — only nodes at this level
        node = queue.popleft()
        # process node
        for nbr in neighbours(node):
            queue.append(nbr)
    depth += 1
```

When to prefer level-snapshot: when the **per-level** computation
matters (e.g., LC #102 Binary Tree Level Order Traversal — you
must group nodes by level in the output; LC #199 Right Side View
— you want the *last* node at each level; LC #1162 — count cells
added at the final non-empty level).

When to prefer triple-with-depth: when you only care about the
final depth, or about per-node distances. Today's #994 is the
final-depth case.

### 4. Python's arbitrary-precision int as a competitive-programming superpower

GCD2 is a single bullet in a longer list of CodeChef problems
where Python's bignum collapses the apparent difficulty:

| Problem | C/C++ trick | Python equivalent |
|---------|-------------|--------------------|
| GCD2 | Read 250-digit number as string, compute `mod small` digit-by-digit | `int(s)`, `math.gcd(a, b)` |
| FCTRL2 (Small Factorials) | Manual array of digits, school-multiplication algorithm for `n! * (n+1)` | `math.factorial(n)` |
| Large fibonacci | Matrix exponentiation with big-int multiplication helpers | `a, b = b, a + b` |
| Modular exponentiation | Repeated squaring | `pow(a, b, m)` — built in |
| Combinations of large numbers | Pascal's triangle in a big-int array | `math.comb(n, k)` |

The catch: Python is ~30-100× slower than C++ per operation, so
problems with **tight TLs and 10^7+ operations** still need C++
even with the bignum savings. But for problems where the *hard
part* is the bignum and the *operation count* is moderate, Python
wins on developer time by orders of magnitude.
