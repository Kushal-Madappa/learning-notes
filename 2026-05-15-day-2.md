## Key concepts

### 1. Set membership vs. dict lookup
Yesterday Two Sum used a dict because we needed to remember **where** we saw
a value. Contains Duplicate only needs to know **whether** we saw it, so a
`set` is the right tool. Rule of thumb:

- Need the associated data (index, count, parent, ...) ⇒ `dict`.
- Just need "have I seen this?" ⇒ `set`.

Both give O(1) average lookups, but a set is leaner.

### 2. Hashing by a canonical fingerprint
Group Anagrams is the canonical "bucket by category" problem. The key idea:
**design a function that maps equivalent inputs to the same hashable key**,
then group with a dict.

- Sorted-letters key: `"".join(sorted(s))` — O(k log k) per string.
- Letter-frequency tuple: `tuple(counts)` — O(k) per string with a fixed
  26-letter alphabet, slightly faster on long strings.

You'll re-use this pattern for problems like "group by sum of digits",
"group collinear points by slope", etc. Whenever the question is "put
things that are equivalent under some relation into the same bucket",
write a fingerprint function and hash on it.

### 4. `int(str(n)[::-1])` is the cleanest reverse-a-number trick
The cast to int automatically drops leading zeros. Worth knowing the
pure-arithmetic version too (`result = result * 10 + n % 10`) for when
string slicing isn't available or when you want digit-level control.
