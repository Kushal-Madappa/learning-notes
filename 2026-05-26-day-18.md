## Today's work

- **LeetCode (1):**
  - **Subtree of Another Tree (#572, Easy)** — given two binary trees
    `root` and `subRoot`, return True iff some node in `root` has a
    subtree that is **structurally and value-wise identical** to
    `subRoot`. The clean decomposition is **nested recursion**:
    - *Outer* DFS walks every node `n` in `root` and asks "does the
      subtree at `n` equal `subRoot`?"
    - *Inner* DFS is **literally LC #100 Same Tree** — value match
      plus pairwise child match.
    - **Why nested and not a single DFS:** Subtree identity is a
      whole-shape property; it can't be decided from O(1) work per
      outer node. The inner recursion has to walk both candidate
      subtrees in lockstep. That's the giveaway shape: *one recursion
      delegates a "is this whole thing equal to that whole thing"
      question to a second recursion.*
    - **The trap:** "Subtree" in this problem means *complete subtree
      from some node down*, not "any connected subgraph." So at each
      outer node we compare the **entire** subtree rooted there — we
      don't get to pick which children to include. The inner helper
      enforces this automatically because both sides must end at
      `None` at the same time; one side ending early is a structural
      mismatch.
    - **Complexity:** O(m * n) time worst case (inner Same Tree is
      O(m) per outer node, and outer is O(n)); O(h_root + h_sub)
      space for the two stacks. Hashing / Merkle / serialization
      gets you O(m + n) but adds machinery — not worth it on LC
      Easy, and the nested-recursion shape is the lesson here.
    - Verified: 2 LC samples + 8 hand-built edges (single-equal,
      single-diff, subRoot==root, deep leaf, values-match-shape-
      doesnt, match-on-right, negatives, subRoot-too-big). All 10
      pass.

- **CodeChef (1):**
  - **Sums in a Triangle (SUMTRIAN, Easy)** — a triangle with row i
    holding i+1 numbers; start at the top, descend, at each step go
    either directly below or one place to the right; maximize the
    total. The first DP problem on the profile.
    - **Why naïve recursion explodes:** Each cell `(i, j)` has two
      children `(i+1, j)` and `(i+1, j+1)`. A pure recursion explores
      *all* root-to-leaf paths, and with up to 99 rows that's
      2^98 paths — about 3 × 10^29. The total cell *count* though is
      only 1+2+…+100 = 5050. Massive overlap → memoize.
    - **The bottom-up rewrite (the one I shipped):** walk rows from
      bottom to top; replace each cell in-place with `cell + max(child_left,
      child_right)`. After one full pass `tri[0][0]` holds the answer.
      Each cell is touched O(1) times → O(n²) overall, where n is the
      number of rows.
    - **Why bottom-up beats top-down memoization here:** Both are O(n²),
      but bottom-up avoids Python's recursion-stack overhead and the
      `lru_cache` decorator overhead. For a competitive-programming
      problem with `t ≈ 1000` test cases and `n ≤ 99`, that constant
      factor matters — top-down can squeak through but flirts with TLE
      in pure Python; bottom-up is comfortable.
    - **The classic "greedy fails" trap:** A natural-but-wrong instinct
      is "always pick the bigger child as you descend." It fails on
      triangles like `[1] / [5 9] / [99 1 1]`: greedy picks
      1 → 9 → 1 = 11, but the optimum is 1 → 5 → 99 = 105 — you had to
      pick the *smaller* child on row 1 to reach the 99. DP doesn't
      care about local greediness; it computes the true best from
      every cell.
    - **Complexity:** O(n²) time per test case, O(1) extra space
      (mutating the triangle in place). For up to 1000 tests × 99
      rows × 99 cols ≈ 10^7 operations — comfortable.
    - Verified: official sample (5 then 9) + 3 hand-built edges
      (single-row, all-zeros, greedy-fails-hairpin, max-size 99-row
      all-99s → 9801). All 4 pass.

## Key concepts

### 1. Nested recursion — "outer walks, inner compares"
A new template lands today, sitting next to the four trees-week
templates already in the toolkit:

| Template           | Shape                                                  | Example         |
|--------------------|--------------------------------------------------------|-----------------|
| Pure-value         | helper returns a scalar                                | #104, #543      |
| Pure-mutation      | helper mutates in place                                | #226            |
| Two-tree           | helper takes *two* nodes, returns bool/count           | #100, #101      |
| Hybrid             | helper returns a value AND mutates outer state         | #543, #110-ish  |
| **Nested**         | **outer DFS walks one tree; at each node, calls a *different* DFS** | **#572**        |

Nested recursion is the unlock for any "find a structure inside a
structure" question. Once you see it:

- **#572 Subtree of Another Tree** — outer walks `root`, inner is
  Same Tree on `(node, subRoot)`.
- **#437 Path Sum III** — outer walks every node, inner counts
  downward paths summing to `targetSum`. (Or the prefix-sum trick to
  collapse to one pass — but nested is the natural first solution.)
- **#652 Find Duplicate Subtrees** — outer walks every node,
  inner serializes the subtree; equal serializations → duplicate.
- **#236 LCA of a Binary Tree** — *not* nested (single DFS suffices),
  but it's worth noticing the family resemblance: ancestor questions
  often look nested but collapse.

The cost is in the multiplication: O(outer × inner) = O(n × m) for
#572. When that becomes the bottleneck, the escape hatch is
**hashing the subtree shape** (Merkle / canonical serialization), so
the inner comparison drops to O(1) lookup → O(n + m) total. Pay the
machinery cost only when the input warrants it.

### 2. The DP recipe — overlapping subproblems → memoize → tabulate
SUMTRIAN is the prototype for the next phase. The recipe to keep
in mind for every DP-shaped problem:

1. **State.** What's the smallest tuple `(i, j, …)` that lets you
   describe a subproblem? For SUMTRIAN: `(i, j)` — "best path sum
   starting at cell `(i, j)`." That's it.
2. **Transition.** How does state `s` decompose into smaller states?
   `best(i, j) = tri[i][j] + max(best(i+1, j), best(i+1, j+1))`.
3. **Base case.** Where does the recursion bottom out? Bottom row:
   `best(n-1, j) = tri[n-1][j]`.
4. **Overlap check.** Do different recursion paths re-ask the same
   `(i, j)`? Yes — `(2, 1)` is reachable from `(1, 0)` and `(1, 1)`,
   etc. → memoize.
5. **Top-down vs. bottom-up.** Top-down = recursion + `@lru_cache`,
   easy to write, stack overhead. Bottom-up = nested loops over
   states in dependency order, no stack, often O(1) extra space.
   Pick bottom-up when the dependency order is obvious (here:
   "rows from bottom to top"); pick top-down when it isn't (e.g.
   memoized graph DP with weird traversal).

This is the mental scaffold for the upcoming DP wave: **#70 Climbing
Stairs** (the canonical 1-D DP), **#198 House Robber** (1-D with
"skip" decision), **#322 Coin Change** (unbounded knapsack), **#1143
Longest Common Subsequence** (2-D). All of them fit this 5-step
recipe.

### 3. "Greedy fails" — the canonical counterexample
SUMTRIAN's `[1] / [5 9] / [99 1 1]` is the cleanest illustration of
why DP isn't just "be smart locally." A greedy descent that always
picks the bigger child gets 11. The actual optimum is 105. The 9 is
a *trap*: it locks you out of the 99 underneath the 5. The lesson
generalizes:

> Greedy is correct only when the locally best choice is provably
> compatible with the global optimum. When a smaller local choice
> can *enable* a larger payoff later, greedy fails — and DP is the
> right tool.

This is the same reason greedy fails on **#322 Coin Change** with
denominations `[1, 3, 4]` and target 6 (greedy: 4+1+1=3 coins;
optimum: 3+3=2 coins). Memorize the pattern: small local sacrifice,
big global win → DP, not greedy.

### 4. In-place table mutation — when it's safe and cheap
The SUMTRIAN solution mutates the input triangle directly:

```python
tri[i][j] += max(tri[i+1][j], tri[i+1][j+1])
```

That's safe because the row-i update only *reads* row i+1, which was
finalized in the previous iteration of the outer `for i` loop. The
dependency graph is a strict DAG: row i depends only on row i+1.
This pattern recurs constantly in DP:

- **Rolling-array trick:** when state `dp[i]` depends only on
  `dp[i-1]`, you only need two rows (or sometimes one) instead of
  the full O(n²) table. SUMTRIAN already uses this implicitly —
  the "table" *is* the input triangle.
- **Same-array updates:** safe iff you respect the dependency
  order. Get it wrong and you read a half-updated value.

Worth carrying into next week: every DP problem, ask "can I drop a
dimension by mutating in place / using a rolling array?" before
allocating a full O(n²) or O(n × W) table.
