## Key concepts

### 1. The two binary-search templates — pick one, stay consistent
Every binary-search bug I've seen comes from mixing the two
templates inside the same function. Memorise them as a single unit:

| Template          | Loop          | Init                 | Update on "discard mid" | Update on "keep mid as candidate" |
|-------------------|---------------|----------------------|-------------------------|-----------------------------------|
| Closed `[lo, hi]` | `while lo<=hi`| `hi = len(nums)-1`   | `lo=mid+1` / `hi=mid-1` | (return immediately on `==`)      |
| Half-open `[lo,hi)`| `while lo<hi` | `hi = len(nums)`     | `lo = mid+1`            | `hi = mid`                        |

Mental model:
- **Closed** is the "yes/no, exact hit" template — it's for
  "find this value or report it's missing." The `lo > hi` exit
  state means "interval is empty, no hit."
- **Half-open** is the "leftmost-true" template — it's for
  "find the boundary where some predicate flips from false to
  true." The `lo == hi` exit state means "the boundary is here."

### 2. Why `hi = len(nums)` (not `len(nums) - 1`) in the half-open template
This is the subtle part of #35. If `target` is larger than every
element, the answer is `n` (insert at the end). With a closed
interval that index is unreachable — `hi` would max out at `n-1`.
The half-open template makes the off-by-one disappear by *defining*
the interval to include the "one past the end" slot.

### 3. `mid = lo + (hi - lo) // 2`, always
In Python it's just a stylistic choice — ints don't overflow. In
C / C++ / Java, `(lo + hi) / 2` can overflow when both are near
`INT_MAX`. Writing the safe form is muscle memory that ports cleanly
the day I'm in another language for a contest.

### 4. Trial division — the √n cutoff
The reason we stop at `i*i <= n` is the contrapositive of a basic
fact: if `n = a*b` with `1 < a ≤ b`, then `a ≤ √n`. So any
composite has a divisor `≤ √n`; finding none means primality.

Two micro-optimisations layered on top:
- Handle 2 explicitly, then step by 2 — halves the iterations.
- (More aggressive, not used here) the 6k±1 trick — step by 2 and
  4 alternately, skipping all multiples of 3 too. Worth it for
  hot loops; overkill for `N ≤ 10⁴`.

### 5. `i*i <= n` vs `i <= sqrt(n)`
The integer-only form (`i*i <= n`) avoids floating-point and the
`int(math.sqrt(n))` rounding traps (e.g. `sqrt(25) == 4.999...`
under some implementations). Same complexity, strictly safer.
File this with last week's "scale into integers when exactness
matters" rule.

### 6. AMR15A and the strict-inequality habit
The output is **"READY FOR BATTLE" iff evens > odds** — strictly
greater, not `>=`. Tie goes to NOT READY. This is the kind of
detail that turns a 30-second problem into a wrong-answer
submission if you skim past the `strictly` in the statement.
General rule: every time the problem says "strictly", "at least",
"at most" or "more than", re-read that clause once more before
coding.

## Patterns to spot next
- **Find first/last position of a target (LC #34)** — apply
  `lower_bound` for the first, `upper_bound` (predicate
  `nums[mid] > target`) for the last. Two binary searches, one
  problem.
- **First Bad Version (LC #278)** — the purest leftmost-true
  problem. The predicate is given as a black-box `isBadVersion(i)`;
  the template doesn't change.
- **Search in Rotated Sorted Array (LC #33)** — binary search
  where the array is split into two sorted halves. The template
  stays; the case-split is: at every step, decide which half is
  sorted, then check if `target` lies in that half.
- **Parametric / "binary search the answer"** — Koko Eating
  Bananas (#875), Capacity to Ship Packages (#1011). The array
  being searched is the *answer space*, and the predicate is
  "can we do it in ≤ k units at this rate?" Same template, very
  different feel.
