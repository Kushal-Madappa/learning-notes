## Key concepts

### 1. Opposite-ends two pointers
The core idea: maintain two indices `left` and `right` that bracket a
candidate region, then move whichever endpoint deterministically
improves the situation.

- **Valid Palindrome.** Skip non-alphanumeric characters inward from
  each side, lower-case-compare. The "skip" loops are bounded by the
  outer condition `left < right`, so the total work is still O(n).
- **Two Sum II.** A sorted array gives you a monotone sum:
  - `nums[L] + nums[R] < target`  =>  must increase L
  - `nums[L] + nums[R] > target`  =>  must decrease R
  No dict needed — sortedness encodes the "have I seen the complement?"
  question for free.

### 2. Why two pointers beats brute force
Brute force pair search is O(n²). Two pointers turns it into O(n)
because each step *eliminates* a row or column of the implicit pair
matrix. That elimination only works because the input has structure
(sorted, palindromic, monotone, etc.). When you can't justify an
elimination step, two pointers won't fit — fall back to hashing.

### 3. The `isalnum`/`lower` toolkit for string problems
Three Python string predicates are worth memorising cold:

| Method        | What it does                                        |
|---------------|-----------------------------------------------------|
| `c.isalnum()` | True for `[A-Za-z0-9]` (plus Unicode letters/digits)|
| `c.isalpha()` | True for letters only                                |
| `c.isdigit()` | True for digits only                                 |

Pair with `.lower()` for case-insensitive comparison. For interview-
grade speed, pre-filter into a list once if you need multiple passes.

### 4. Buffered I/O — the FLOW002 / START01 lesson
Both CodeChef problems today reuse the same scaffold from Day 2:

```python
data = sys.stdin.read().split()
# parse from `data`, accumulate output, then sys.stdout.write once.
```

`split()` with no argument splits on any whitespace, which lets the
same code handle space- or newline-separated input. The `_test()` in
each script proves the parser using `io.StringIO`, so the file is
self-testing without needing real stdin.

## Patterns to spot next
- **Sliding window** — once both pointers move in the same direction
  and you're tracking a running aggregate, you've moved from two
  pointers into sliding window. Watch for "longest / shortest
  substring with property X".
- **Fast / slow pointers** — same family, different problem class:
  detect cycles, find midpoint, in-place rewrites.
- **3Sum and friends** — sort, then fix one index and two-pointer the
  rest. Day 4 candidate.
