## Key concepts

### 1. Parametric binary search has the same skeleton every time
Koko (#875), Ship Packages (#1011), Aggressive Cows, and even
Find-the-Smallest-Divisor (#1283) are all the same five-line
template:

```python
def feasible(k):  # the predicate; True iff k is good enough
    ...

lo, hi = <lower bound>, <upper bound>
while lo < hi:
    mid = (lo + hi) // 2
    if feasible(mid):
        hi = mid
    else:
        lo = mid + 1
return lo
```

Today the only invention was the predicate
`hours(k) = Σ ⌈p / k⌉ ≤ h`. Everything else is muscle memory from
Day 8/9/10. The mental model: **"sort the answer space by 'good
enough?' and lower_bound the True region."** That's it.

Why this works: `hours(k)` is monotonically non-increasing in `k`
(faster eating never *needs* more hours). So the False/True boundary
exists and is unique, and `lower_bound` finds it.

### 2. `ceil(a, b)` without floats: `(a + b - 1) // b`
The single most reusable identity from today. Floating-point
`math.ceil(p / k)` is correct in Python but fragile across languages,
and even in Python it costs an unnecessary FP conversion. The
integer form `(a + b - 1) // b` works for all non-negative `a`,
positive `b`, and is exact in any fixed-width integer type as long
as `a + b - 1` doesn't overflow.

This pairs naturally with Day 10's `mid > x // mid` for sqrt: both
are members of the same family — "rewrite the math so it stays
inside integers." If I had to keep three idioms in muscle memory
forever, this is one of them.

### 3. CodeChef style rule: when `input()` becomes the bottleneck
The config (added 2026-05-20) says: use plain `input()`; reach for
`sys.stdin.readline` / `buffer.read()` only when the problem is
specifically about fast I/O — e.g. INTEST. **FCTRL is in that
bucket.** The constraint `T ≈ 10⁵` plus a one-line solution means
~99% of the runtime is reading input, and stock `input()` on
10⁵ lines is on the edge of TLE on CodeChef's Python judge.

Heuristic I'm adopting:

| Condition                                        | I/O choice           |
|--------------------------------------------------|----------------------|
| `T ≤ 10⁴` and constant work per line             | `input()`            |
| `T ≥ 10⁵`, *or* total integers read ≥ 10⁶        | `sys.stdin.readline` |
| Problem explicitly tagged "fast I/O" (INTEST)    | `sys.stdin.buffer.read().split()` |

This keeps the "minimal style" the config asks for, while still
clearing the judge.

### 4. CONFLIP — why parity wins over simulation
The naive solve is "simulate N rounds, flipping a length-N array
each time." That's O(N²) and TLEs once `N` hits 10⁵ or so. The
observation that makes it O(1):

> In round `r`, every coin with index ≤ r flips. So coin `k`
> participates in rounds `k, k+1, …, N`, i.e. it flips
> `N − k + 1` times.

Parity is all that matters for the final face, and we only care
about *how many* coins land each way, not *which* ones. So we count
how many indices `k ∈ [1, N]` have `N − k + 1 ≡ 0 (mod 2)`. That's
exactly `⌊N / 2⌋`. Same-face count = `N // 2`; opposite-face count
= the rest.

This is the same "find the invariant, throw away the simulation"
muscle that MUFFINS3 used yesterday. Forming it as a reflex:

> *"Is there a quantity I can count without running the process?"*

### 5. Lapindrome: multisets beat sort-and-compare
For "do these two strings have the same characters with the same
multiplicities?" the three standard tools are:

1. `sorted(left) == sorted(right)` — O(N log N), trivial to write.
2. `Counter(left) == Counter(right)` — O(N), cleanest in Python.
3. Fixed-size frequency array of 26 ints — O(N) and tiny constant
   factor; what you'd write in C++.

I used (2) because Python's standard library has `Counter` and the
constraint (string of lowercase letters) is small. In a tighter
language I'd write (3). The naming matters: in interviews, the
phrase "compare as multisets" describes the operation correctly,
which beats "sort both and check equality" as a 1-line spoken
explanation.

## Patterns to spot next

- **Capacity to Ship Packages Within D Days (#1011, Medium)** — the
  binary-search twin of Koko. Predicate becomes `can_ship(capacity)`:
  walk the weights, start a new day whenever the running sum
  exceeds the candidate capacity, return True iff days ≤ D. The
  lower bound is `max(weights)` (any capacity below that can't even
  fit the heaviest package), upper bound is `sum(weights)` (ship
  everything in one day).
- **Search a 2D Matrix (#74, Medium)** — flatten conceptually:
  treat the `m × n` matrix as a sorted 1-D array of length `m * n`,
  half-open binary search over `[0, m * n)`, decode the picked
  index as `row, col = mid // n, mid % n`. Same engine.
- **Find Peak Element (#162, Medium)** — first non-monotonic
  predicate: compare `nums[mid]` to `nums[mid + 1]`. If `nums[mid]
  < nums[mid + 1]`, a peak must lie to the right (or be `mid + 1`
  itself); else, a peak lies at `mid` or left. Trains the muscle of
  using binary search on a *local* condition, not a global
  monotone-predicate.
